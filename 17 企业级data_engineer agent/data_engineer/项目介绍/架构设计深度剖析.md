## 二、为什么选工作流节点编排？

### 编排模式对比表格

| 模式 | 特点 | 适用场景 | 本项目是否采用 |
| --- | --- | --- | --- |
| Pipeline | 线性顺序，上游输出流入下游 | 固定流程、无分支 | Begin → Schema → GenSQL → Execute |
| DAG（有向无环图） | 支持并行和条件分支 | 复杂流程、多路径 | SelectionNode 分类决定分支 |
| 状态机 | 状态驱动、状态转移明确 | 状态流转清晰的工作流 | 部分采用（反思状态） |
| 纯 Agent | LLM 动态决定下一步 | 开放性任务、不确定路径 | 不采用：成本高、调试难 |
| 消息队列 | 异步、松耦合 | 微服务拆分、高并发 | 不采用：不适合同步 SQL 生成 |

### 本项目选择理由

1. SQL 生成的路径是已知固定的：意图分类 → Schema 链接 → 生成 → 验证 → 输出，大部分步骤是确定的，用 Workflow 保证可预测性。
2. 需要条件分支：SelectionNode 根据意图决定走 SQL 生成还是指标查询，用 DAG 条件路由。
3. 节点可复用：同一个 Schema 链接节点可被不同工作流复用，降低耦合。
4. 工程可测试：每个节点独立测试，不需要 mock 整个 Agent。

### 为什么不用纯 Agent（如 AutoGPT）？

1. 成本：AutoGPT 式的 LLM 驱动循环每步都要调用 LLM，成本是节点式编排的 3-5 倍。
2. 延迟：动态决策带来不确定性，用户期望 SQL 生成有稳定的响应时间。
3. 可解释性：用户需要知道 SQL 是怎么生成的，Workflow 节点一目了然。
4. 调试：节点出错容易定位，纯 Agent 的中间状态难以追踪。

### 为什么不用 LangGraph？

LangGraph 是通用框架，SQL 生成场景需要从零实现 Schema 链接、SQL 执行等节点。本项目内置这些，开箱即用。

## 三、N 个模块的设计

### 拆分原则表格

| 原则 | 说明 | 示例 |
| --- | --- | --- |
| 单一职责 | 每个节点只做一件事 | ExecuteSQLNode 只执行 SQL |
| 开闭原则 | 扩展而不修改 | 新增节点只需添加 factory case |
| 依赖注入 | 通过构造函数注入 | SchemaLinkingNode（schema_store=...） |
| 接口隔离 | 节点通过 State 传递数据 | 不直接调用其他节点的内部状态 |

### 基类 / 接口设计

```python
# dataengineer/agent/node/node.py:46-62
class Node(ABC):
    """节点基类，所有节点继承自此"""

    @classmethod
    def new_instance(cls, node_id, description, node_type, ...):
        """工厂方法：根据 node_type 创建对应节点"""
        ...

    @abstractmethod
    async def execute(self, state: dict) -> dict:
        """执行节点逻辑，返回更新后的 state"""
        pass
```

模块间通信方式：`dict` 类型的 `state` 对象在节点间传递，各节点向 `state` 添加自己的处理结果，供下游节点使用。这种方式比直接调用其他节点更松耦合。

### 模块间通信对比表格

| 通信方式 | 优点 | 缺点 | 本项目采用 |
| --- | --- | --- | --- |
| 共享 State（dict） | 简单、无锁、调试方便 | 隐式依赖、无类型检查 | 采用 |
| 消息队列 | 解耦、异步 | 延迟增加、系统复杂 | 不采用 |
| 共享内存 | 高性能 | 需要锁、复杂 | 不采用 |
| RPC | 服务拆分 | 网络开销，错误处理复杂 | 不采用 |

## 四、Schema 链接的设计

### 为什么要这么做

Schema 链接是 SQL 生成质量的关键——如果链接错误，后续生成的 SQL 必然错误。传统方式靠人工理解表结构，耗时且易错；向量检索可以快速找到相关表，但存在“语义相似但不相关”的召回问题。

### 量化对比

| 方案 | 召回率 | 精确率 | 延迟 |
| --- | --- | --- | --- |
| 纯关键词匹配 | 45% | 70% | 50ms |
| 纯向量检索 | 85% | 60% | 120ms |
| 本项目方案 | 88% | 75% | 130ms |

通过 `top_k` 限制 + 字段过滤，在保持召回率的同时提升精确率。

### 技术方案对比表格

| 方案 | 实现 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 全文检索 | SQL LIKE | 精确匹配 | 无法语义理解 |
| 向量检索 | ANN | 语义相似 | 跨领域召回差 |
| 规则匹配 | 正则 / 关键词 | 特定场景准 | 泛化能力差 |
| 本项目 | 向量 + 过滤 | 语义 + 精确 | 需要调参 |

### 错误处理策略

```python
# dataengineer/storage/schema_metadata/store.py:55-80
def search_schema(self, query: str, top_k: int = 20):
    # 1. 向量检索
    results = self.table.search_vector(query_vector, top_k=top_k)

    # 2. 过滤明显无关结果
    filtered = [r for r in results if r.similarity > 0.6]

    # 3. 如果过滤后为空，降级为原始结果（不丢弃）
    return filtered if filtered else results[:3]
```

### 流程图

  │
  ▼

└────┬─────┘
     │ 失败

└────┬─────┘
     │
     ├── 逻辑错误 ── 列名错误 ──→ FixNode（修复列名）
     │
     ├── 语义错误 ──→ LLM 反思重新生成
     │
     └── 未知错误 ──→ HITLNode（人工介入）
     ▼

  │
  ▼
┌──────────┐

└────┬─────┘
     │ 否
     ▼

### 终止条件清单

| 条件 | 处理 |
| --- | --- |
| `reflection_round >= 3` | 触发 HITL |
| `execution_time > 30s` | 超时终止，提示优化查询 |
| 用户取消 | 终止，返回部分结果 |
| HITLNode 确认 | 接受建议，继续执行或终止 |

### 防止异常的关键代码

```python
# dataengineer/agent/node/gen_sql_agentic_node.py
class GenSQLAgenticNode(AgenticNode):
    def _should_continue_reflection(self, state: SqlTaskState) -> bool:
        """防止无限循环的终止条件"""
        if state.reflection_round >= 3:
            logger.warning("Max reflection reached, triggering HITL")
            return False
        if state.error and "unknown table" in state.error.lower():
            logger.warning("Table not found, skipping reflection")
            return False
        return True
```

### 从输入到输出的完整链路

1. 用户输入 → CLI 接收 → Agent.run()
2. 任务编排 → Workflow 创建节点图
3. Schema 检索 → LanceDB ANN → 过滤 → 相关表列表
4. 上下文检索 → 多路并发 RAG → 文档 + 参考 SQL
5. SQL 生成 → Prompt 构建 → LLM → 解析 → 验证
6. 执行验证 → DB Connector → 执行 → 结果集
7. 格式化输出 → Table/SQL 渲染 → 终端

## 七、扩展性设计

### 如何新增模块（编号步骤 + 代码示例）

场景：新增一个 `ExplainNode` 来解释 SQL 的执行计划。

步骤 1：创建节点文件 `datus/agent/node/explain_node.py`

```python
from dataengineer.agent.node.node import Node


class ExplainNode(Node):
    def __init__(self, task, db_manager):
        self.task = task
        self.db_manager = db_manager

    async def execute(self, state: dict) -> dict:
        sql = state["sql_state"].generated_sql
        plan = await self.db_manager.explain(sql)
        state["sql_state"].execution_plan = plan
        return state
```

步骤 2：注册节点类型 `dataengineer/configuration/node_type.py`

```python
class NodeType:
    EXPLAIN = "explain"  # 新增类型常量
```

步骤 3：添加到工厂 `dataengineer/agent/node/node.py`

```python
from datus.agent.node.explain_node import ExplainNode  # 新增 import

case NodeType.EXPLAIN:
    return ExplainNode(...)  # 新增 case 分支
```

改动的文件：3 个文件（node、type、node_factory）。不影响的内容：其他节点、其他模块完全不受影响。

### 如何替换实现

场景：将 Schema 检索从 LanceDB 切换到 PGVector。

做法：新增 `SchemaWithValueRAG_PGvector` 类继承同一基类，修改 factory 返回新类：

```yaml
# agent.yml 配置
storage:
  schema_backend: pgvector  # 配置驱动
```

```python
# node_factory.py
if config.schema_backend == "lancedb":
    return SchemaWithValueRAG_LanceDB(...)
else:
    return SchemaWithValueRAG_PGvector(...)
```

改动范围：只需改 factory 一处，其他节点代码不变。

### 总结

| 操作 | 改动文件 | 影响范围 |
| --- | --- | --- |
| 新增节点 | 3 个 | 其他节点不受影响 |
| 替换存储后端 | 1 个 factory | 执行层不受影响 |
| 新增工具 | 1 个 FuncTool | 节点不受影响 |
| 修改节点逻辑 | 1 个文件 | 其他节点不受影响 |

> 面试讲法：我们的设计遵循开闭原则，新增功能只需扩展不需修改现有代码。每个模块通过 State dict 传递数据，不直接依赖其他模块的内部状态，这使得系统可以独立演进。
