## 📌 项目简介

Data_engineer 是一个数据工程智能体（Data Engineering Agent），通过领域感知推理、语义搜索和持续学习，将自然语言转换为准确的 SQL 语句。

### 核心功能列表

- 🧠 **自然语言转 SQL**：NL → 意图理解 → 语义检索 → SQL 生成
- 📊 **Evoluble Context**：构建活的知识库，捕获 schema 元数据、参考 SQL、语义模型、指标和领域知识
- 🔄 **Subagent 封装**：将成熟的领域封装为 scoped chatbot，提供 API/Web/MCP 多种交付方式
- 📈 **语义层 & 指标**：通过 MetricFlow 集成定义业务指标 YAML，生成跨数据库方言的 SQL
- 🔌 **MCP 协议**：既是 MCP Server（暴露工具给 Claude Desktop、Cursor），也是 MCP Client
- 🛠 **Skills 系统**：支持 agentskills.io 风格的打包工具、可配置权限和市场支持
- 🗄 **多数据库支持**：内置 SQLite/DuckDB + 11 种可插拔数据库适配器
- 🤖 **多 LLM 提供者**：支持 OpenAI、Claude、Gemini、DeepSeek、Qwen、Kimi 等 10+ 提供者

### 项目类型

- 开发语言：Python 3.12+
- 技术栈：OpenAI Agents SDK + LiteLLM, LanceDB, FastAPI, FastMCP, Streamlit
- 状态：生产可用，活跃维护

        self.tools = {}  # 可用工具

        self.metadata_store = None  # Schema 元数据
        self.metrics_store = None  # 指标存储

位置：`datus/agent/agent.py`

### 2. Workflow（工作流编排）

职责：管理节点有向图、执行顺序、上下文传递。

```python
class Workflow:
    nodes: Dict[str, Node]  # 节点映射
    node_order: List[str]  # 执行顺序
    context: Context  # 工作流上下文
    status: str  # pending/running/completed/failed/paused

    def add_node(self, node, position=None):
        """添加节点到工作流"""
```

位置：`datus/agent/workflow.py`

### 3. Node（节点基类）

职责：定义节点接口，子类实现具体逻辑。

```python
class Node(ABC):
    @abstractmethod
    def execute(self, state: dict) -> dict:
        """执行节点逻辑"""
        pass

    @classmethod
    def new_instance(cls, node_id, description, node_type, ...):
        """工厂方法：根据 node_type 创建对应节点"""
```

## 🔄 核心流程：SQL 生成流程

### 流程图

```text
用户输入: "查询每月营收 TOP 10 的产品"
   │
   ├── BeginNode
   │   (初始化)
   │
   └── SelectionNode
       (意图分类)
          │
          ▼
   SchemaLinkingNode
       (表结构检索)
          │
          ▼
   GenSQLAgenticNode
       (生成 SQL + 反思)
          │
          ├── [SQL 合理] ──→ ExecuteSQLNode ──→ OutputNode
          │                                      (输出结果)
          │
          └── [SQL 存疑] ──→ FixNode ──→ ReflectNode
                                (修复)      (反思)
                                   │
                                   └── [仍失败?] ──→ HITLNode
                                                       (人工介入)
```

### 关键决策点

1. 意图分类：区分 SQL 生成、指标查询、文档搜索等
2. Schema 链接：确定需要哪些表和字段
3. SQL 生成：基于上下文和 Schema 生成 SQL
4. 反思验证：检查 SQL 语法和逻辑正确性
5. 执行验证：实际执行 SQL 验证结果

## 🐛 常见问题与调试

### Q1：SQL 生成结果为空

症状：输入查询后返回空结果。

原因：Schema 链接失败，未找到相关表。

解决：

```python
# 检查表是否在上下文中
agent.context.schema_info.get("available_tables")

# 手动触发 Schema 检索
agent.run_node("schema_linking", force=True)
```

### Q2：LLM 调用超时

症状：请求超时或响应慢。

原因：模型负载高或网络问题。

解决：

```bash
# 切换更快的模型
datus config set target.model gpt-4.1-mini
```

### Q3：向量检索无结果

症状：文档搜索返回空。

原因：平台文档未初始化。

解决：

```python
from datus.storage.document.doc_init import init_platform_docs

init_platform_docs(platform="duckdb", version="latest")
```

### Q4：数据库连接失败

症状：无法连接到数据源。

原因：连接配置错误。

解决：

```bash
# 检查配置
datus config show

# 验证连接
datus-cli --datasource demo /Check SELECT 1
```

### Q5：MCP 工具不可见

症状：MCP Client 连接后看不到工具。

原因：MCP Server 未正确配置。

解决：

```bash
# 检查 MCP 配置
cat .datus/config.yml | grep mcp

# 重启 MCP 服务
datus-mcp restart
```

## 📊 性能优化建议

| 瓶颈 | 当前方案 | 优化方向 | 优先级 |
| --- | --- | --- | --- |
| Schema 检索延迟 | 全表扫描 | 预计算 embedding index | 高 |
| LLM 生成耗时 | 单次调用 | 增加缓存/批量处理 | 高 |
| 向量检索精度 | 纯向量 | 混合 BM25 + rerank | 中 |
| Session 存储 | SQLite | 切换到更快的 KV 存储 | 中 |
| 多节点执行 | 串行 | 并行/流水线执行 | 低 |
