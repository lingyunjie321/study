## 一、项目一句话介绍

Data_engineer 是一个面向数据工程场景的自然语言转 SQL 智能体平台。它可以把用户的自然语言问题转成可执行 SQL，并结合数据库 schema、参考 SQL、业务文档和指标定义来提升生成准确率。

如果面试官让你用 30 秒介绍，可以这样说：

> 这个项目是一个数据工程智能体，核心能力是自然语言转 SQL。用户输入一个业务问题后，系统会先理解意图，再检索相关表结构和业务上下文，然后调用大模型生成 SQL，最后执行 SQL 并根据执行结果做自动修复。我的重点工作是把这个流程拆成可控的工作流模块，并优化了 schema 检索、混合检索和 SQL 反思修复机制。

## 二、面试表达原则

### 1. 不展示代码时，少说这些

面试口述时尽量少说内部实现名，例如：

- 某某 Node 类
- 某某 Result 对象
- 某某 factory 方法
- 某某配置变量
- 某某私有函数
- 具体文件路径

这些词会让回答听起来像背源码，而且面试官没有代码上下文时也不容易接住。

### 2. 面试时多说这些

更推荐使用下面这种表达：

| 代码化表达 | 面试口语表达 |
|---|---|
| 某某节点 | 一个独立处理阶段 / 一个工作流模块 |
| 工厂方法创建节点 | 统一的节点创建器按类型生成模块 |
| 状态对象传递 | 把任务状态在各阶段之间传递 |
| 向量库查询方法 | 在向量库里做语义相似度检索 |
| 反思节点和修复节点 | 失败分析模块和 SQL 修复模块 |
| 最大反思轮次变量 | 最多自动修复几轮，避免无限循环 |
| schema store | schema 元数据存储 |
| document store | 文档知识库存储 |

### 3. 推荐回答结构

回答项目问题时，尽量按这个顺序：

1. 先讲业务目标：这个模块解决什么问题。
2. 再讲数据流：输入是什么，中间怎么处理，输出是什么。
3. 再讲技术取舍：为什么这样设计。
4. 最后讲效果：准确率、延迟、自动修复率、成本等。

## 三、简历口径

### 项目背景

数据工程师在日常工作中经常需要根据业务问题写 SQL，例如统计营收、订单、留存、转化率等。传统方式依赖人工理解表结构、字段含义和业务口径，效率低，而且跨数据库方言时容易出错。

这个项目的目标是构建一个数据工程智能体，让用户通过自然语言提出问题，系统自动完成 schema 检索、上下文补充、SQL 生成、执行验证和错误修复，降低人工写 SQL 的成本。

### 主要负责内容

可以写成下面这种简历表达：

- 设计并实现自然语言转 SQL 的工作流架构，将 SQL 生成拆分为意图理解、schema 链接、上下文检索、SQL 生成、执行验证和反思修复等阶段，提升系统可控性和可扩展性。
- 构建基于向量检索的 schema 链接能力，将表结构、字段说明、样例值等元数据向量化存储，支持从大量表中快速召回相关表和字段。
- 设计混合检索策略，将语义向量检索和关键词检索结合，并加入字段加权、结果去重和多样性控制，提升检索准确率。
- 实现 SQL 自动反思修复机制，根据语法错误、字段不存在、表选择错误、执行超时等失败类型选择不同修复策略，减少人工介入。
- 支持多种数据库和多种大模型供应商，并通过统一配置和适配层降低接入成本。

### 量化结果表达

如果你准备在简历或面试中使用量化指标，可以这样说：

- 混合检索策略使 benchmark 检索准确率从 0.71 提升到 0.89。
- Top-5 召回准确率从 65% 提升到 82%。
- 通过预计算向量索引，schema 检索延迟从秒级降低到百毫秒级。
- 反思修复机制可以覆盖大部分常见 SQL 语法错误和字段错误。
- 支持 10+ 大模型供应商和 11 种数据库适配。

注意：面试时如果被追问指标来源，可以解释 benchmark 是固定测试集上的评测结果，不要说得像线上绝对指标。

## 四、STAR 项目讲述

### S：背景

数据工程团队经常需要根据自然语言需求编写 SQL。人工写 SQL 的问题是效率低、依赖个人对表结构和业务口径的理解，而且跨数据库方言时容易出错。

例如同样是“统计每月营收 Top 10 商品”，不同数据库的时间函数、聚合语法、分页语法都可能不一样。如果表结构复杂，单靠人工查 schema 也很耗时。

### T：任务

我的目标主要有四个：

1. 把自然语言问题自动转成可执行 SQL。
2. 解决大量表结构下的 schema 召回问题。
3. 通过业务文档、参考 SQL、指标定义补充上下文。
4. 建立 SQL 生成后的验证和修复机制，提高生成稳定性。

### A：行动

我没有把所有逻辑放进一个大模型调用里，而是把流程拆成多个阶段：

```text
用户问题
→ 意图理解
→ Schema 链接
→ 上下文检索
→ SQL 生成
→ SQL 执行
→ 失败反思与修复
→ 结果输出
```

在 schema 链接上，我把表结构、字段说明、样例值等元数据向量化存储。用户问题进来后，系统先做语义检索，召回相关表和字段，而不是把全量 schema 都塞给大模型。

在检索优化上，我结合了向量检索和关键词检索。向量检索解决语义相似问题，关键词检索解决表名、字段名、SQL 关键词这类精确匹配问题。最后再通过字段加权和多样性控制重新排序，避免结果都集中在同一类文档或同一张表上。

在 SQL 生成上，我会把相关 schema、参考 SQL、业务文档和用户问题一起组织成 prompt，让模型在有限上下文中生成 SQL。

在反思修复上，SQL 生成后不会直接返回，而是先执行或校验。如果失败，就根据数据库返回的错误信息判断失败类型，再选择重新检索 schema、修复语法、调整字段、重新生成 SQL 等策略。同时设置最大自动修复轮次，防止无限循环。

### R：结果

最终系统在 benchmark 上的检索准确率从 0.71 提升到 0.89，Top-5 准确率从 65% 提升到 82%。Schema 检索延迟从秒级降低到百毫秒级。常见 SQL 语法错误和字段错误可以通过反思机制自动修复，减少了人工介入。

## 五、核心架构问答

### Q1：这个项目整体架构是什么？

可以这样回答：

> 这个项目整体是一个自然语言转 SQL 的数据工程智能体。用户从 CLI、API 或 MCP 入口输入问题后，系统会进入任务编排层。任务编排层会按固定流程调用几个模块：先理解用户意图，再做 schema 链接，找出相关表和字段；然后检索参考 SQL、业务文档和指标定义；接着调用大模型生成 SQL；最后执行 SQL，并根据执行结果决定是否需要修复。
>
> 我把这个系统拆成几个阶段，而不是写成一个大模型黑盒。这样做的好处是每个阶段职责清晰，出了问题也容易定位。比如 SQL 里字段不存在，我可以判断是 schema 召回问题还是生成阶段的问题；如果 SQL 能执行但结果不对，就进入反思修复流程。

面试官想听的点：

- 你能说清数据流
- 你能解释为什么分层
- 你不是只会“调大模型”

### Q2：为什么要拆成工作流，而不是一个大 Agent？

可以这样回答：

> 因为 SQL 生成场景的大部分步骤是确定的，比如 schema 链接、上下文检索、SQL 生成、执行验证、失败修复，这些步骤不需要每一步都让大模型自由决定。用工作流可以保证路径可控、延迟稳定、便于调试。
>
> 如果用纯 Agent 模式，灵活性会更强，但问题是每一步都可能调用模型做决策，成本更高、执行路径更不可控，出错后也很难定位。我们这个项目更偏工程生产场景，所以选择结构化工作流加局部智能决策的方式。

一句话总结：

> 固定流程用工作流保证可控，复杂判断交给大模型处理，这样兼顾稳定性和智能能力。

### Q3：你们的节点管理用了工厂模式，工厂模式是什么？

可以这样回答：

> 工厂模式是一种创建对象的设计模式。简单说，就是业务流程不用自己关心每个模块怎么创建，而是把创建逻辑交给一个统一的“工厂”。业务流程只需要说“我要一个 schema 检索模块”或者“我要一个 SQL 执行模块”，工厂根据类型创建对应模块。
>
> 在我们项目里，工作流会根据配置里的模块类型，统一创建 schema 链接、SQL 生成、执行验证、结果输出等模块。这样新增模块时，比如新增指标查询模块或文档搜索模块，只需要增加对应实现并注册类型，不需要大改主流程。

工厂模式的价值：

- 创建逻辑集中管理
- 主流程不依赖具体模块实现
- 新增模块更方便
- 配置驱动工作流更自然

面试时不要展开成源码，只讲思想即可。

### Q4：新增一个功能模块怎么扩展？

可以这样回答：

> 这个项目的扩展方式是新增一个独立处理阶段，然后把它挂到工作流里。比如我要新增 SQL 解释功能，可以新增一个“SQL 解释模块”，输入是生成后的 SQL 和执行结果，输出是自然语言解释。主流程不用重写，只需要在工作流配置中把这个阶段加进去。
>
> 这种设计符合开闭原则：原有模块尽量不改，新增能力通过扩展模块实现。

## 六、Schema 链接与检索问答

### Q5：Schema 链接具体怎么做？

可以这样回答：

> Schema 链接就是从大量数据库表里找出和用户问题相关的表和字段。这个项目里不是把所有表结构都交给模型，而是先把表名、字段名、字段类型、表描述、DDL 信息和部分样例值整理成可检索的元数据，并向量化存储。
>
> 用户输入问题后，系统会把问题也转成向量，在向量库里检索相似 schema，同时结合数据库名、schema 名、表类型等过滤条件，召回一批候选表。最后只把相关表和字段放进 SQL 生成上下文里。

流程：

```text
用户问题
→ 问题向量化
→ 检索表结构和样例值
→ 过滤候选结果
→ 控制候选数量
→ 返回给 SQL 生成阶段
```

### Q6：为什么要检索样例值？

可以这样回答：

> 只看表名和字段名，有时召回不准。比如用户问“已支付订单”，字段可能叫 status，表描述里也不一定写“已支付”，但是样例值里可能有 paid、success 这样的枚举值。把样例值也纳入检索，可以补充业务语义，提升 schema 召回准确率。

### Q7：表太多时怎么筛？

可以这样回答：

> 我主要从三层控制：
>
> 第一，用候选数量限制，只取最相关的一批表，而不是全量塞给模型。
>
> 第二，用数据库名、schema 名、表类型等结构化条件过滤，避免跨库、跨业务域误召回。
>
> 第三，用字段相关性和样例值相关性做二次判断，避免只因为表描述相似就把不相关表放进去。

### Q8：如果向量检索召回不准怎么办？

可以这样回答：

> 我会从四个方向优化。
>
> 第一，优化分块策略。schema、文档和参考 SQL 要按语义单元组织，不能简单按固定长度切开。
>
> 第二，换更适合场景的 embedding 模型。中文自然语言和英文表字段混合的场景，通用模型不一定最优，可以选择中文和中英混合效果更好的模型。
>
> 第三，引入混合检索。向量检索擅长语义匹配，但对表名、字段名、SQL 关键词这种精确匹配不一定稳定，所以需要结合关键词检索。
>
> 第四，做二次排序和多样性控制。初步召回后重新排序，并限制同一文档或同一类结果占据太多位置。

一句话总结：

> 不是只依赖向量检索，而是用语义召回、关键词召回、字段加权、二次排序和多样性控制一起提升准确率。

### Q9：混合检索为什么比单纯向量检索好？

可以这样回答：

> 单纯向量检索的问题是容易语义漂移。比如 SQL 关键词、表名、字段名，本身就需要精确匹配，向量模型可能觉得语义相似但实际业务不相关。
>
> 所以我把向量检索和关键词检索结合起来。向量检索负责理解用户问题的语义，关键词检索负责命中具体表名、字段名、标题和 SQL 术语。最后再按权重融合，整体准确率会更稳定。

## 七、SQL 生成与反思问答

### Q10：SQL 是怎么生成的？

可以这样回答：

> SQL 生成不是直接把用户问题丢给大模型，而是先准备上下文。上下文主要包括相关表结构、字段说明、样例值、参考 SQL、业务文档和指标定义。然后把这些信息和用户问题一起组织成 prompt，让模型生成 SQL。
>
> 生成后还会做解析、执行和验证。如果执行成功，就返回 SQL 和结果；如果失败，就进入反思修复。

### Q11：反思机制是自动触发的吗？

可以这样回答：

> 是自动触发的。用户正常使用时不需要手动点击“反思”。触发点在 SQL 生成之后、最终输出之前。系统会先尝试执行或校验生成的 SQL，如果发现失败，就自动进入反思流程。

触发条件包括：

- SQL 语法错误
- 表不存在
- 字段不存在
- SQL 方言不兼容
- 执行超时
- 查询结果明显异常
- SQL 结构和用户意图不匹配

触发后的流程：

```text
生成 SQL
→ 执行或校验
→ 成功则输出
→ 失败则分析错误原因
→ 选择修复策略
→ 重新生成或修复
→ 再次执行
```

### Q12：反思机制具体怎么修？

可以这样回答：

> 反思不是简单让模型“再生成一次”，而是先分类错误，再选策略。
>
> 如果是语法错误，就把数据库返回的错误信息和原 SQL 一起给模型，让它修复语法。
>
> 如果是字段不存在，说明可能 schema 召回不完整或者字段选错，就重新检查相关表字段。
>
> 如果是表不存在，优先重新做 schema 召回。
>
> 如果 SQL 能执行但结果不符合问题，就让模型结合用户问题、SQL 和执行结果重新判断逻辑，比如是否缺少分组、排序、过滤条件或 join 关系。

### Q13：如何防止反思无限循环？

可以这样回答：

> 系统会限制最大自动修复轮次，比如最多 3 轮。每次反思都会记录当前轮次和失败原因。如果达到上限，就不会继续自动重试，而是返回错误信息、进入更保守的推理流程，或者交给人工处理。
>
> 这个限制很重要，因为 LLM 修复不是百分百可靠。工程系统不能无限消耗 token 和时间，必须有明确的退出条件。

### Q14：如果所有反思都失败怎么办？

可以这样回答：

> 如果自动修复多轮后仍然失败，系统不会一直重试，而是降级处理。比如返回当前 SQL、错误原因和可能的问题方向，让用户补充更明确的业务口径，或者转人工处理。
>
> 我觉得这也是工程系统应该有的边界：不是所有问题都追求自动解决，关键是失败时要可解释、可定位、可恢复。

### Q15：SQL 生成最大的难点是什么？

可以这样回答：

> 最大难点不是调用大模型，而是让 SQL 生成稳定、可控、可验证。
>
> 一方面，如果 schema 召回错了，模型基本不可能生成正确 SQL。另一方面，即使 schema 对了，模型也可能编造字段、误解业务口径或用错数据库方言。
>
> 所以我的思路是用检索保证上下文质量，用执行验证发现错误，用反思修复覆盖常见问题，同时用最大轮次限制控制成本和风险。

一句话总结：

> 这个项目最大的难点，是在复杂 schema 和不稳定大模型输出之间，构建一条可检索、可生成、可验证、可修复的 SQL 生成链路。

## 八、框架选型问答

### Q16：为什么不用 LangGraph 或 CrewAI？

可以这样回答：

> 我不是认为这些框架不好，而是这个项目的主流程比较明确，更适合轻量工作流。
>
> SQL 生成的链路大致是固定的：用户问题、schema 链接、上下文检索、SQL 生成、执行验证、反思修复、输出。大部分阶段不需要每一步都让 Agent 自由决策。
>
> CrewAI 更适合多个角色型 Agent 协作，比如研究员、规划员、执行员互相通信。但这个项目的核心不是多角色讨论，而是稳定地生成和验证 SQL。如果引入多 Agent，反而会增加角色通信、状态同步和调试成本。
>
> LangGraph 更适合复杂状态图和动态分支场景。我们这个项目当然也有分支和反思，但主要诉求是可控、可测试、可插拔，所以自研一层轻量工作流更贴合业务。

一句话总结：

> 不是为了不用框架而不用，而是因为这个场景流程明确，自研轻量工作流更可控、调试成本更低。

### Q17：什么是 MetricFlow 流程？

可以这样回答：

> MetricFlow 可以理解成语义层查询工具。它解决的问题是：业务指标不能每次都让大模型临时理解，否则口径容易不一致。
>
> 比如“营收”这个指标，业务上可能要扣除退款、优惠、取消订单。如果每次让模型自由生成 SQL，可能今天一种口径，明天另一种口径。MetricFlow 的思路是提前定义好指标、维度、时间粒度和过滤规则。用户问指标类问题时，系统先识别指标和维度，再通过语义层生成标准 SQL。

流程：

```text
用户问题
→ 识别指标、维度和时间范围
→ 匹配语义层里的指标定义
→ 生成标准 SQL
→ 执行并返回结果
```

一句话总结：

> MetricFlow 主要解决指标口径一致性问题，让“营收”“活跃用户”“订单数”这类指标由语义层统一定义，而不是完全依赖大模型临时发挥。

### Q18：为什么需要 MCP？

可以这样回答：

> MCP 是一种让 AI 应用和外部工具标准化交互的协议。对这个项目来说，它的价值是把 SQL 生成、文档检索、数据库查询等能力暴露给外部工具，比如 IDE 或桌面 AI 助手。
>
> 这样用户不用专门打开我们的系统，也可以在支持 MCP 的环境里通过自然语言调用这些能力。

面试时可以补一句：

> 我理解 MCP 的价值不只是“让模型调用工具”，更重要的是统一工具发现、参数传递和调用协议，降低不同 AI 应用之间的集成成本。

## 九、工程化与性能问答

### Q19：向量检索性能瓶颈在哪里？

可以这样回答：

> 主要有三个瓶颈。
>
> 第一是 embedding 生成，如果每次请求都实时生成大量向量，会影响延迟。
>
> 第二是向量库检索，数据量小的时候直接检索问题不大，数据量大时需要建索引。
>
> 第三是结果后处理，比如过滤、去重、排序和拼接上下文。
>
> 我的优化思路是：schema 和文档提前向量化，查询时只生成用户问题向量；向量库提前建索引；检索后只保留最相关的候选结果，避免把大量无关上下文交给模型。

### Q20：如何降低大模型 token 消耗？

可以这样回答：

> Token 消耗主要来自 prompt 里的 schema、文档和参考 SQL。我的策略是只放必要上下文，不把全量 schema 都塞进去。
>
> 具体来说，先通过 schema 链接筛出相关表，再只放相关字段；参考 SQL 只选最相关的几条；文档也只放和问题强相关的片段。简单问题可以用较小模型，复杂问题再路由到能力更强的模型。

一句话总结：

> 先检索，再压缩，再生成，避免把大模型当全文搜索器用。

### Q21：如果文档规模扩大到百万级，怎么扩展？

可以这样回答：

> 百万级文档时，单机向量库和本地处理可能会成为瓶颈。我会从四层扩展。
>
> 第一，向量存储从单机方案迁移到分布式向量库，比如 Milvus 或云上的向量检索服务。
>
> 第二，文档解析和向量化改成异步任务队列，多 worker 并行处理。
>
> 第三，API 服务做水平扩展，前面加负载均衡。
>
> 第四，引入缓存，把热点 query、热点 schema 和常用指标结果缓存起来。

### Q22：如何保证代码质量和系统稳定？

可以这样回答：

> 我主要从测试、配置、日志和降级四个方面保证。
>
> 测试方面，核心模块要有单元测试，比如检索、SQL 解析、数据库适配、反思修复。
>
> 配置方面，模型、数据库、工具权限都通过配置管理，避免写死在代码里。
>
> 日志方面，每次任务都记录关键阶段，比如检索耗时、生成耗时、SQL 执行结果、失败原因。
>
> 降级方面，如果自动修复失败，系统不会无限重试，而是返回可解释错误或转人工处理。

### Q23：密钥和配置怎么管理？

可以这样回答：

> API Key 和数据库连接信息不能写死在代码里，一般放在环境变量或本地配置文件里。配置文件只保留变量占位符，运行时从环境变量读取。这样可以区分开发、测试和生产环境，也避免密钥泄露。

## 十、补充高频技术八股

这一部分不是每场面试都会问，但如果面试官偏基础或偏系统设计，很容易从这些方向追问。回答时仍然保持“概念 + 项目联系 + 取舍”的结构。

### Q24：这个项目是否涉及 harness？怎么理解？

可以这样回答：

> 涉及。这里的 harness 可以理解成给大模型套上的一层工程化外壳。大模型本身是不稳定的，如果完全让它自由决定下一步，路径会不可控，也很难调试。
>
> 在这个项目里，大模型不是独立乱跑，而是被放进一个固定的 SQL 生成流程里：先检索 schema，再补充上下文，再生成 SQL，再执行验证，再反思修复。每一步都有输入、输出、错误处理和终止条件。
>
> 所以我理解的 harness，就是把大模型能力放进一个可控、可观测、可回滚的工程框架中，让它在边界内发挥作用。

一句话总结：

> 这个项目不是让 LLM 自由行动，而是用工作流、工具、权限、日志和反思机制把它约束在可控链路里。

### Q25：Agent、Workflow、Multi-Agent 有什么区别？

可以这样回答：

> Agent 更强调自主决策，模型会根据中间结果决定下一步做什么，适合开放性任务。
>
> Workflow 是预先定义好的流程，步骤相对固定，适合 SQL 生成这种路径明确的任务。
>
> Multi-Agent 是多个专业 Agent 分工协作，比如一个负责检索、一个负责规划、一个负责执行，适合复杂任务拆分。
>
> 这个项目更接近“工作流 + 局部 Agent 能力”。主流程是确定的，但在 SQL 生成、错误分析、反思修复这些环节，会用大模型做智能判断。

### Q26：Pipeline、DAG、状态机有什么区别？

可以这样回答：

> Pipeline 是线性流水线，上一步输出给下一步，比如检索、生成、执行、输出。
>
> DAG 是有向无环图，比 Pipeline 更灵活，可以支持分支和并行，但不能形成无限循环。
>
> 状态机是根据当前状态决定下一步，比如 SQL 执行成功就输出，执行失败就进入反思，反思超过上限就停止。
>
> 这个项目的主链路像 Pipeline，但反思修复和意图分流会带一点 DAG 和状态机思想。

### Q27：ReAct、CoT、Reflection 怎么理解？

可以这样回答：

> CoT 是让模型按步骤推理，适合复杂理解问题。
>
> ReAct 是边推理边行动，比如模型先分析，再调用工具，再根据工具结果继续分析。
>
> Reflection 是模型对已有结果做检查和修正。
>
> 这个项目里最明显的是 Reflection：SQL 生成后，如果执行失败，系统会把错误信息、用户问题和原 SQL 结合起来分析，再决定是修语法、重查 schema，还是重新生成。

### Q28：Tool Calling / Function Calling 是什么？

可以这样回答：

> Tool Calling 是让大模型不只是输出自然语言，而是输出结构化的工具调用请求。比如它判断需要查数据库，就生成一个带参数的数据库查询请求；判断需要搜索文档，就生成一个文档检索请求。
>
> 它的好处是可以让大模型连接外部能力，比如数据库、文件系统、搜索工具、指标系统。
>
> 风险是模型可能生成错误参数，所以工具层必须做参数校验、权限控制和错误处理，不能完全相信模型输出。

### Q29：RAG 是什么？为什么适合这个项目？

可以这样回答：

> RAG 就是检索增强生成。模型回答之前，先从外部知识库里检索相关内容，再基于检索结果生成答案。
>
> 这个项目里，RAG 很适合用来补充 schema、业务文档、参考 SQL 和指标定义。因为这些知识经常变化，不能完全依赖模型训练时的记忆。
>
> 通过 RAG，系统可以基于当前数据库和当前业务文档生成 SQL，减少模型凭空编造。

### Q30：RAG 和微调有什么区别？

可以这样回答：

> RAG 是在推理时补充上下文，不改变模型参数；微调是用训练数据改变模型行为。
>
> 如果知识经常变化，比如表结构、业务指标、文档说明，RAG 更合适，因为更新知识库就能生效。
>
> 如果目标是让模型学习固定输出格式、固定风格或特定领域表达，微调更合适。
>
> 这个项目里 schema 和业务文档会变化，所以优先用 RAG，而不是把所有知识都微调进模型。

### Q31：RAG 的幻觉问题怎么解决？

可以这样回答：

> RAG 不能完全消除幻觉，因为幻觉可能来自两个地方：第一是检索结果不相关，第二是模型没有严格基于检索内容生成。
>
> 我的解决思路是三层控制。
>
> 第一，提升检索质量，用混合检索、二次排序和多样性控制减少无关上下文。
>
> 第二，生成阶段约束模型尽量基于给定 schema 和参考 SQL，不要编造字段。
>
> 第三，生成后通过 SQL 执行验证，如果字段或表不存在，就触发反思修复。

### Q32：Chunking 为什么重要？

可以这样回答：

> Chunking 决定检索系统的上限。切得太碎，会丢上下文；切得太大，会带入太多噪声。
>
> 对这个项目来说，文档分块要尽量保留完整语义，比如一个 SQL 示例、一个表定义、一段指标说明最好不要被切散。
>
> 所以我更倾向于按标题、段落、SQL 示例、schema 单元做语义分块，而不是简单按固定字符数切。

### Q33：Embedding 模型怎么选？

可以这样回答：

> Embedding 模型选择要看语言、领域和成本。
>
> 如果是中文问题加英文表字段的混合场景，要选对中文和中英混合检索效果比较好的模型。
>
> 如果追求低成本和低延迟，可以选轻量模型；如果追求高准确率，可以选更大的 embedding 模型，但存储和检索成本会更高。
>
> 所以不是模型越大越好，而是要结合准确率、延迟和成本做取舍。

### Q34：Rerank 是怎么做的？

可以这样回答：

> Rerank 就是初步召回后再排序。第一阶段召回主要保证不要漏，第二阶段排序主要保证最相关的排前面。
>
> 这个项目里可以用轻量 rerank，比如结合向量相似度、关键词命中、标题命中、字段名命中、层级路径命中等信号综合打分。
>
> 如果对准确率要求更高，也可以引入更重的模型做二次排序，但会增加延迟。

### Q35：GraphRAG 或知识图谱在这个项目里有什么价值？

可以这样回答：

> 纯向量检索擅长语义相似，但不擅长表达结构关系。数据库场景里，表和表之间的 join 关系、主外键关系、业务实体关系都很重要。
>
> 如果引入图结构，可以把表、字段、指标、业务实体之间的关系建成图，用来辅助多跳推理。比如用户问“客户复购”，系统可能需要从用户表、订单表、支付表之间找到正确 join 路径。
>
> 所以 GraphRAG 的价值是补足向量检索对结构化关系理解不足的问题。

### Q36：并行执行时状态冲突怎么解决？

可以这样回答：

> 并行处理时尽量避免多个线程或任务同时修改同一份状态。
>
> 我的思路是让每个并行任务返回自己的独立结果，最后在主流程里统一合并。比如多个文档并行解析，每个任务只返回自己的解析结果，不直接写全局状态。
>
> 如果确实需要更新共享统计信息，就要用锁或原子操作保护，避免计数错误或状态覆盖。

### Q37：缓存一致性怎么处理？

可以这样回答：

> 缓存的核心问题是更新后不能读到旧数据。对于 schema、文档这类数据，我会记录版本或更新时间。如果检测到内容变化，就删除旧缓存并重新写入。
>
> 对热点查询结果，可以设置过期时间，避免长时间返回过期结果。
>
> 另外，写入时要避免一边更新一边被读取到半成品数据。可以采用先写新版本，再切换引用的方式，或者先删旧数据再批量写入新数据。

### Q38：Schema 演进怎么处理？

可以这样回答：

> Schema 演进就是数据库表结构变化，比如新增字段、删除字段、字段类型变化。
>
> 如果 schema 变了，但向量库里还是旧信息，模型就可能生成不存在的字段。所以需要定期或触发式同步 schema。
>
> 我的做法是扫描当前数据库元数据，和已存版本做对比。如果发现变化，就重新生成对应表的检索内容和向量索引。

### Q39：如何做评估？

可以这样回答：

> 评估不能只靠主观感觉，要分层看。
>
> 检索层看召回率、准确率、Top-K 命中率。
>
> SQL 生成层看语法正确率、执行成功率、结果正确率。
>
> 性能层看 schema 检索延迟、端到端响应时间、token 消耗。
>
> Benchmark 就是用固定测试集重复评估，方便比较不同检索策略、prompt 或模型带来的变化。

### Q40：日志和可观测性怎么设计？

可以这样回答：

> 我会记录每次请求在关键阶段的状态，比如 schema 检索耗时、召回了哪些表、SQL 生成耗时、SQL 执行是否成功、失败原因是什么、是否触发反思。
>
> 日志级别上，正常流程用 info，非致命异常用 warning，真正失败用 error。
>
> 这样出问题时可以快速判断是检索问题、生成问题、数据库执行问题，还是外部模型服务问题。

### Q41：熔断和降级怎么做？

可以这样回答：

> 如果外部模型服务、数据库或工具服务连续失败，系统不能一直重试到资源耗尽，需要有熔断和降级。
>
> 熔断是失败率过高时暂时停止调用某个服务，快速失败。
>
> 降级是当某个能力不可用时，使用备用策略。比如文档检索不可用时，只基于 schema 生成；大模型不可用时返回可解释错误；自动修复失败时转人工。

### Q42：项目如何部署到生产环境？

可以这样回答：

> 生产环境可以拆成 API 服务、MCP 服务、向量存储、数据库连接和日志监控几部分。
>
> API 服务负责对外提供自然语言查询能力，MCP 服务负责接入外部 AI 工具，向量存储负责 schema 和文档检索，数据库连接层负责执行 SQL。
>
> 真正上线前，还需要补齐权限控制、SQL 审计、查询超时、资源隔离、敏感字段保护和监控告警。

### Q43：这个项目和同类项目有什么区别？

可以这样回答：

> 我觉得区别在于它不是单纯的聊天机器人，而是围绕数据工程场景做了完整链路。
>
> 第一，有 schema 链接，能从大量表里找相关表字段。
>
> 第二，有执行验证，生成 SQL 后会实际校验，而不是只返回文本。
>
> 第三，有反思修复，能根据错误类型自动调整。
>
> 第四，支持指标语义层和多数据库适配，更接近真实数据平台场景。

## 十一、开放追问回答

### Q44：如果面试官质疑你只是复现项目，怎么回答？

可以这样回答：

> 我不是只跑通项目，而是重点理解和优化了 SQL 生成链路里的几个关键问题。
>
> 第一，我梳理了从用户问题到 SQL 输出的完整工作流，理解每个阶段为什么要拆开。
>
> 第二，我重点优化了检索部分，把单纯向量检索扩展成混合检索，提升 schema 和文档召回质量。
>
> 第三，我梳理并完善了 SQL 失败后的反思修复逻辑，把错误按语法、字段、schema、逻辑几类处理，而不是简单重试。
>
> 所以我对这个项目的理解不是停留在“能跑”，而是知道它为什么这样设计、瓶颈在哪里、怎么优化。

### Q45：如果重新设计，你会怎么改？

可以这样回答：

> 如果重新设计，我会重点改三点。
>
> 第一，把检索层抽象得更统一，让 schema、文档、参考 SQL、指标定义都走统一的检索接口，方便后续切换向量库。
>
> 第二，把评估体系做进 CI 或日常回归里，让每次改检索权重、prompt 或模型后，都能自动跑 benchmark，看准确率有没有退化。
>
> 第三，把反思策略做得更可配置，比如不同数据库、不同业务场景可以配置不同修复策略，而不是所有错误都走同一套流程。

### Q46：项目最大的技术难点是什么？

可以这样回答：

> 我认为最大的难点是 SQL 生成质量的稳定性，而不是模型调用本身。
>
> SQL 生成依赖两个前提：一是 schema 召回要准，二是模型不能幻觉。schema 少了会缺表缺字段，schema 多了模型容易混淆。模型本身也可能编造字段、误用函数、理解错业务指标。
>
> 所以这个项目的重点是把生成链路工程化：先用检索控制上下文，再用执行验证发现错误，再用反思修复覆盖常见问题，最后用最大轮次和人工兜底控制风险。

### Q47：这个项目有没有上线？

可以这样回答：

> 可以按你真实情况回答。如果只是本地或实验项目，不要硬说线上生产。
>
> 保守说法是：这个项目目前主要以本地 CLI、API 服务和 MCP 工具形式运行，用于验证自然语言转 SQL、schema 检索和反思修复链路。它具备生产化部署的基础，比如配置化模型和数据库、多数据库适配、日志记录和错误处理。如果要上线，还需要补充权限控制、审计、监控和更完整的回归测试。

### Q48：Benchmark 是什么？

可以这样回答：

> Benchmark 可以理解成固定的评测集或标准测试流程，用来衡量系统效果。比如我们用一批固定问题和标准答案来评估 schema 召回、SQL 执行成功率和生成准确率。
>
> 我说准确率从 0.71 提升到 0.89，意思是在同一套测试集上，优化前后的检索或生成效果有明显提升。它不是随便拍脑袋的线上感觉，而是基于固定评测样本算出来的。

## 十二、面试高频短句

### 架构类

- 这个项目不是一个单次大模型调用，而是一条可控的 SQL 生成流水线。
- 固定流程交给工作流，复杂判断交给大模型。
- 拆成多个阶段的好处是解耦、可测试、可扩展。
- SQL 错了以后能定位是 schema 召回问题、生成问题，还是执行问题。

### 检索类

- Schema 链接是 SQL 生成质量的前置条件。
- 不把全量 schema 塞给模型，而是先检索再压缩。
- 向量检索解决语义相似，关键词检索解决精确匹配。
- 样例值能补充字段名之外的业务语义。
- Rerank 的目标是让更相关、更覆盖多样性的结果排在前面。

### 反思类

- 反思不是简单重试，而是先分类错误，再选修复策略。
- 反思机制是自动触发的，触发点在 SQL 执行或校验失败后。
- 最大反思轮次是为了防止无限循环和 token 浪费。
- 工程系统不能追求无限自动修复，要有失败边界和降级方案。

### 框架类

- 不用 LangGraph 或 CrewAI，不是因为它们不好，而是这个场景流程明确，轻量工作流更可控。
- CrewAI 更适合多角色协作，我们这个项目更偏确定性 SQL 生成流水线。
- MetricFlow 解决的是业务指标口径一致性问题。
- 工厂模式就是把模块创建逻辑集中管理，让主流程不依赖具体实现。

### 难点类

- 最大难点是 SQL 生成质量稳定性，而不是调用大模型。
- Schema 召回和 LLM 幻觉是两个核心风险。
- 这个项目的价值在于把不稳定的大模型输出放进一个可验证、可修复的工程闭环里。

## 十三、1 分钟完整项目回答模板

> 这个项目是一个面向数据工程场景的自然语言转 SQL 智能体。用户输入业务问题后，系统会先理解意图，再做 schema 链接，从大量表里找出相关表和字段；然后检索参考 SQL、业务文档和指标定义作为上下文；接着调用大模型生成 SQL；最后执行 SQL，并根据执行结果判断是否需要自动修复。
>
> 我主要负责的是整体 SQL 生成链路的工作流设计、schema 检索优化和反思修复机制。架构上我没有让大模型自由决策所有步骤，而是把流程拆成多个可控阶段，这样更容易调试、测试和扩展。
>
> 检索上，我用了向量检索加关键词检索的混合策略。向量检索负责语义理解，关键词检索负责表名、字段名、SQL 关键词这类精确匹配，再通过加权排序和多样性控制提升召回质量。
>
> 生成后，系统会自动执行或校验 SQL。如果出现语法错误、字段不存在、表选错、执行超时等问题，会自动进入反思修复流程。反思不是无限重试，而是按错误类型选择不同修复策略，并设置最大轮次防止死循环。
>
> 最终在 benchmark 上，混合检索把准确率从 0.71 提升到 0.89，schema 检索延迟也从秒级优化到百毫秒级。这个项目最大的难点是如何在复杂 schema 和大模型幻觉之间，构建一条可检索、可生成、可验证、可修复的 SQL 生成链路。

## 十四、2 分钟深入项目回答模板

> 这个项目的背景是数据工程师经常要根据业务问题写 SQL，但是人工写 SQL 依赖对表结构、字段含义和业务口径的理解，效率比较低，而且跨数据库方言时容易出错。所以我做的是一个数据工程智能体，把自然语言问题转成可执行 SQL。
>
> 整体架构上，我把它拆成了几个阶段：用户问题进入系统后，先做意图理解，判断是普通 SQL 查询、指标查询还是文档查询；如果是 SQL 查询，就先做 schema 链接，从大量表里召回相关表和字段；然后检索参考 SQL、业务文档、指标定义等上下文；接着调用大模型生成 SQL；生成后不是直接返回，而是先执行或校验；如果失败，就进入反思修复；最后输出 SQL 和结果。
>
> Schema 链接是这里的关键，因为表选错了，后面 SQL 一定错。我没有把全量 schema 都塞给模型，而是把表名、字段、描述、DDL 和样例值都整理成可检索的元数据。用户问题进来后，先做向量检索召回相关 schema，再用数据库名、schema 名、表类型和候选数量限制过滤结果。
>
> 单纯向量检索有语义漂移问题，所以我又加了混合检索。向量检索负责语义相似，关键词检索负责精确匹配，尤其是表名、字段名和 SQL 关键词。之后再结合标题、层级路径、关键词命中等信息做加权排序，并限制同一来源结果过度集中。
>
> SQL 生成阶段，我会把相关 schema、参考 SQL、文档片段和用户问题组织成 prompt。生成后系统会执行 SQL，如果执行失败，就根据错误类型走不同修复策略。比如语法错误就修语法，字段不存在就重新检查 schema，表不存在就重新召回，逻辑不匹配就让模型结合问题和执行结果重新调整。为了避免无限循环，自动修复会限制最大轮次。
>
> 我觉得这个项目最大的难点不是调模型，而是让模型输出变得稳定可控。所以我的设计重点是：检索阶段控制上下文质量，执行阶段发现错误，反思阶段修复常见问题，工程上再用超时和最大轮次控制风险。
