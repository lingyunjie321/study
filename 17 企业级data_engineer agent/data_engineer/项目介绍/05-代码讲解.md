## 1. SqlTaskState — 核心状态模型

**文件**: `data_engineer/schemas/node_models.py`

状态模型是工作流中各节点传递的载体，包含一个 SQL 查询从输入到输出的所有中间状态。

**关键代码片段**：

```python
# 简化后的 SqlTaskState 定义
@dataclass
class SqlTaskState:
    task_id: str                        # 任务唯一 ID
    nl_query: str                      # 用户自然语言查询
    datasource: str                    # 目标数据源

    # 中间结果
    intent: str = ""                   # 意图分类结果
    target_tables: List[TableInfo] = []  # 链接到的表
    generated_sql: str = ""            # 生成的 SQL
    executed: bool = False            # 是否已执行
    result: Optional[QueryResult] = None  # 执行结果

    # 反思相关
    reflection_needed: bool = False   # 是否需要反思
    error: Optional[str] = None       # 错误信息
    reflection_round: int = 0         # 反思轮次
```

**设计要点**: 单一数据类，所有节点共享，通过修改 state 传递处理结果。
**面试考点**: 如果不加 `executed` 字段，如何判断 SQL 是否已执行？

## 2. Node 基类 — 节点抽象接口

**文件**: `data_engineer/agent/node/node.py:46-80`

所有节点都继承自 `Node` ABC，通过工厂方法 `new_instance()` 根据 `node_type` 创建对应节点子类。

**关键代码片段**：

```python
class Node(ABC):
    @classmethod
    def new_instance(cls, node_id, description, node_type, ...):
        """工厂方法：根据 node_type 创建对应节点"""
        from data_engineer.agent.node import (
            BeginNode, SchemaLinkingNode,
            GenSQLAgenticNode, ExecuteSQLNode, ...
        )
        case node_type:
            NodeType.BEGIN:        return BeginNode(...)
            NodeType.SCHEMA_LINK:  return SchemaLinkingNode(...)
            NodeType.GEN_SQL:      return GenSQLAgenticNode(...)
            # ... 其他类型

    @abstractmethod
    async def execute(self, state: dict) -> dict:
        """每个节点必须实现此方法"""
        pass
```

**设计要点**: 工厂模式 + 抽象基类，新增节点只需添加 case 分支，不破坏现有代码。
**面试考点**: 为什么用抽象基类而不是普通基类？——强制子类实现 `execute()` 方法，保证接口一致。

## 3. Workflow — 工作流编排器

**文件**: `data_engineer/agent/workflow.py:19-80`

Workflow 管理节点的生命周期和执行顺序，通过 `add_node()` 添加节点，通过 `node_order` 控制执行拓扑序。

**关键代码片段**：

```python
class Workflow:
    def __init__(self, name, task=None, agent_config=None):
        self.nodes = {}               # node_id → Node 实例
        self.node_order = []           # 执行顺序
        self.context = Context()       # 共享上下文
        self.status = "pending"       # pending/running/completed/failed

    def add_node(self, node: Node, position: int = None) -> str:
        """添加节点到工作流"""
        self.nodes[node.id] = node
        node.workflow = self
        if position is not None:
            self.node_order.insert(position, node.id)
        else:
            self.node_order.append(node.id)

    def execute_all(self):
        """按 node_order 顺序执行所有节点"""
        for node_id in self.node_order:
            node = self.nodes[node_id]
            self.context = node.execute(self.context)
```

**设计要点**: 节点执行顺序由 `add_node` 顺序决定，支持插入位置；状态机管理工作流生命周期。
**面试考点**: Workflow 如何处理节点执行失败？——当前版本继续执行（容错），失败状态记录在 `context.status`。

## 4. SchemaWithValueRAG — Schema 存储检索

**文件**: `data_engineer/storage/schema_metadata/store.py`

Schema 存储通过 LanceDB 实现向量检索，将表名、列名、描述转换为向量，支持语义相似度搜索。

**关键代码片段**：

```python
class SchemaWithValueRAG(BaseEmbeddingStore):
    def search_schema(self, query: str, top_k: int = 20) -> List[Dict]:
        """向量检索相关表结构"""
        # 1. 将查询嵌入为向量
        query_vector = self.embedding_model.embed(query)

        # 2. LanceDB ANN 检索
        results = self.table.search_vector(
            query_vector,
            top_k=top_k,
            where=self._version_filter(),
        )

        # 3. 解析为 TableInfo
        return [self._parse_table(row) for row in results]

    def _parse_table(self, row) -> TableInfo:
        """解析行为 TableInfo 对象"""
        return TableInfo(
            table_name=row["table_name"],
            columns=self._parse_columns(row["columns"]),
            description=row.get("description", ""),
            row_count=row.get("row_count", 0),
        )
```

**设计要点**: 继承 `BaseEmbeddingStore`，复用向量检索能力；`BaseEmbeddingStore` 统一管理 embedding 模型和 LanceDB 连接。
**面试考点**: top_k=20 是如何确定的？——经验值：太少可能漏掉相关表，太多增加后续过滤负担。

## 5. GenSQLAgenticNode — SQL 生成核心

**文件**: `data_engineer/agent/node/gen_sql_agentic_node.py:45-175`

GenSQLAgenticNode 是 SQL 生成的核心节点，内部实现 Prompt 构建、上下文检索、LLM 调用、结果解析的完整流程。

**关键代码片段**：

```python
async def _run_once(self, state: SqlTaskState) -> SqlTaskState:
    """单次生成-验证循环"""

    # 1. 构建 Prompt
    prompt = self._build_prompt(state)

    # 2. 检索相关上下文（文档 + 参考 SQL）
    context = await self._retrieve_context(state)

    # 3. 调用 LLM
    response = await self._call_llm(prompt, context)

    # 4. 解析生成的 SQL
    sql = self._parse_sql(response)

    # 5. 验证
    is_valid, error = self._verify_sql(sql)
    if not is_valid:
        state.reflection_needed = True
        state.error = error
        state.reflection_round += 1

    state.generated_sql = sql
    return state

def _build_prompt(self, state: SqlTaskState) -> str:
    """构建 SQL 生成 Prompt"""
    schema = format_tables(state.target_tables)
    ref_sqls = self._get_reference_sql(state.nl_query)

    return f"""
    给定以下数据库 Schema:
    {schema}

    参考 SQL:
    {ref_sqls}

    问题: {state.nl_query}
    SQL:
    """
```

**设计要点**: 单次循环设计 + 外部循环控制反思次数；Prompt 中注入 Schema 和参考 SQL 提升生成质量。
**面试考点**: 如何防止 Prompt 注入？——Schema 来自内部存储，不来自用户输入，所以安全。

## 6. DocumentStore — 文档向量存储

**文件**: `data_engineer/storage/document/store.py:75-180`

DocumentStore 是平台文档的向量存储实现，继承 `BaseEmbeddingStore`，通过 `search_docs()` 提供语义搜索。

**关键代码片段**：

```python
class DocumentStore(BaseEmbeddingStore):
    TABLE_NAME = "document"

    def search_docs(self, query, version=None, top_n=10, select_fields=None):
        """语义搜索文档"""
        conditions = []
        if version:
            conditions.append(eq("version", version))

        where = And(conditions) if len(conditions) > 1 \
                else conditions[0] if conditions else None

        # 复用父类的 search 方法
        results = self.search(
            query_txt=query,
            top_n=top_n,
            where=where,
            select_fields=select_fields,
        )
        return results.to_pylist()

    def store_chunks(self, chunks: List[PlatformDocChunk]) -> int:
        """存储文档分块（去重后写入）"""
        # 先删除已有 chunk
        chunk_ids = [c.chunk_id for c in chunks]
        for batch in chunk_ids_batch(chunk_ids, 500):
            self.table.delete(in_("chunk_id", batch))

        # 后写入新 chunk
        data = [c.to_dict() for c in chunks]
        self.store_batch(data)
        return len(chunks)
```

**设计要点**: delete-then-add 策略避免 merge_insert 的 commit conflict；分批删除防止 WHERE 子句过长。
**面试考点**: 为什么不用 merge_insert？——merge_insert 在并发时有 commit conflict 问题。

## 7. SemanticChunker — 语义分块

**文件**: `data_engineer/storage/document/chunker/semantic_chunker.py:92-250`

SemanticChunker 实现基于文档结构的智能分块，保留标题层级和段落完整性，避免固定大小切分破坏语义。

**关键代码片段**：

```python
class SemanticChunker:
    def chunk(self, doc: ParsedDocument, base_metadata: dict) -> List[PlatformDocChunk]:
        chunks = []
        for section in doc.sections:
            section_chunks = self._chunk_section(section, titles=[doc.title])
            chunks.extend(section_chunks)

        # 合并过小的 chunk
        chunks = self._merge_small_chunks(chunks)
        return chunks

    def _chunk_section(self, section, titles, base_metadata, start_index):
        """按层级递归分块"""

        # h4+ 层级扁平化
        if section.level >= self.config.max_heading_depth:
            flat_text = self._flatten_section_content(section)
            return self._split_content(flat_text, titles, base_metadata, start_index)

        # 小节整体保留（不超过阈值时）
        total = self._flatten_section_content(section)
        if len(total) <= self.config.chunk_size * self.config.section_merge_buffer:
            return self._split_content(total, titles, base_metadata, start_index)

        # 否则递归处理子节点
        chunks = []
        if section.content:
            chunks.extend(self._split_content(section.content, titles, ...))
        for child in section.children:
            chunks.extend(self._chunk_section(child, titles + [section.title], ...))
        return chunks
```

**设计要点**: 层级优先（h1→h2→h3 保持同区），小节整体保留，过大才拆分；`_flatten_section_content` 递归拼接子节点内容。
**面试考点**: 重叠大小 `chunk_overlap=128` 是怎么来的？——经验值：太小无法保留边界上下文，太大增加检索噪音。
