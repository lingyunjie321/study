## RL 训练流程

每个 step 对应 `RayPPOTrainer.fit()` 中的一段循环，按顺序执行。

### Step 1：训练入口

```text
verl/trainer/main_ppo.py:main()
├── 加载 Hydra 配置
│   ├── scripts/config/ppo_trainer.yaml
│   └── CLI override
└── 初始化 RayPPOTrainer
    └── 创建 Ray WorkerGroup
```

### Step 2：数据加载

- `DataLoader` 读取 `AEARPO/rl_datasets/finance_train.parquet`
- 每步随机采样 `batch_size=128` 条 prompt
- 通过 `chat_template` tokenize，`max_prompt_length=1536`
- padding 后封装为 `DataProto`
- 将 `DataProto` 送入 rollout engine

### Step 3：Rollout：生成与工具调用

```text
vLLMRolloutWithTools.generate(prompts, logprobs=10)
└── SamplingParams
    ├── temperature = 1.0
    ├── top_p = 0.9
    └── max_tokens = 4096
```

生成过程中会不断检查工具调用标签：

- 检测到 `<search>`、`<python>`、`<sandbox>`：调用 `ToolAgent.step()` 执行工具，将结果拼接回上下文并继续生成。
- 未检测到工具标签：检查 EOS、`call_limit=3`、entropy 连续下降等终止条件。
- 触发循环检测：使用 `_detect_tool_loop()` 滑动窗口识别重复调用。
- 计算 entropy：通过 `_calc_entropy()` 得到 token 级 `H_norm`。
- 分支采样：`P(branch) = random() - 0.5 * entropy_delta`，触发后生成 `beam_size` 个共享前缀的分支。

Rollout 返回内容：

```text
trajectory + log_prob(old_log_prob) + entropy + reward
```

### Step 4：Reward 计算

```text
compute_score(trajectory, ground_truth)
├── 数值题：提取最终答案，与 ground_truth 按 ±1% 容差比对，输出 0/1
└── 概念题：基于关键词 overlap 计算 F1，输出 0.0~1.0
```

最终 reward 会写入 `DataProto.batch` 中的 `token_level_rewards`。

### Step 5：Advantage 估计

GRPO 基础公式：

```python
advantage = (reward - group_mean) / (group_std + 1e-8)
```

ARAEPO 可选熵感知调整：

```python
entropy_normalized = (entropy - entropy_mean) / entropy_std
advantages *= 1 + 0.2 * entropy_normalized.detach()
```

输出为形状 `batch_size x sequence_length` 的 `advantage` tensor。

### Step 6：Policy Update

```text
ratio = exp(log_prob - old_log_prob)
PPO loss = -advantages * ratio
```

裁剪逻辑：

- 标准 PPO：`torch.clamp(ratio, 1 - cliprange_low, 1 + cliprange_high)`
- ARAEPO：`max_bound = (1 + cliprange_high) / ratio.detach() * ratio`
- Dual-Clip：防止 `advantage < 0` 时过度惩罚
- KL penalty：可选地加入参考策略约束
- FSDP all-reduce 后执行 `optimizer.step()`
- 收集指标：`pg_loss`、`pg_clipfrac`、`ppo_kl`、`entropy`，并写入 wandb

### Step 7：保存 checkpoint

```text
default_local_dir/global_step_{step}/actor/       # FSDP actor 分片
latest_checkpointed_iteration.txt                # 训练进度
merge_ckpt/convert_checkpoint_from_verl_to_hf_qwen3.sh
```

## 新增金融子领域流程

以新增 `new_domain` 为例：

1. 在 `finance_economics.py` 中新增 `DOM = "new_domain"` 区段。
2. 使用 `add(question, answer, domain, difficulty)` 添加核心题。
3. 在 `templated()` 中添加对应的模板化生成逻辑。
4. 在 `SYSTEM_MSGS` 字典中添加该领域的 system prompt。
5. 运行数据生成脚本，重新生成 Parquet 与 JSONL。
6. 训练时通过 `TRAIN_FILES` 指向新的 Parquet 文件。

```bash
python finance_economics.py
```

## 架构拆分与通信方式

### 分层说明

| 层级 | 组件 | 职责 |
| --- | --- | --- |
| Driver | `RayPPOTrainer` | 协调各阶段执行，控制训练循环 |
| Worker | `ActorRollout`、`Critic`、`RefPolicy` | 执行模型推理、训练、价值评估和参考策略计算 |
| Algorithm | `ARAEPO` 熵平衡机制 | 完成熵感知优势估计和自适应裁剪 |
| Data | `DataProto`、`DataLoader` | 负责数据流转、批处理和协议封装 |

### 编排模式对比

| 模式 | 代表框架 | 优势 | 劣势 | 本项目适用性 |
| --- | --- | --- | --- | --- |
| Pipeline | LangChain | 简单直观 | 难以并行 | 不适合 |
| DAG | AutoGen | 任务拆分灵活 | 调度复杂 | 不适合 |
| 状态机 | ToolAgent | 状态清晰 | 扩展性一般 | 适合作为基础工具调用层 |
| Ray + 混合引擎 | ARAEPO | GPU 感知、容错、推理训练并行 | 配置复杂 | 最佳选择 |

### 选择 Ray + 混合引擎的原因

1. GPU 资源感知：Ray 自动管理 GPU 分配，vLLM 和 FSDP 可以共享 GPU，但显存配置相互独立。
2. 容错机制：Worker 失败后可自动恢复，减少手动重启训练的成本。
3. 推理训练并行：`hybrid_engine` 模式允许 vLLM 生成时，FSDP 准备下一个 step 的梯度计算。
4. 动态负载均衡：`balance_batch()` 可根据序列长度重新分配数据到 GPU。

### 为什么不直接用纯 vLLM

vLLM 是无状态推理引擎，无法直接支持以下训练需求：

- 多轮交互状态追踪
- 工具调用后的策略更新
- 优势估计与策略梯度计算

因此项目需要在 vLLM 基础上封装 `ToolAgent` 状态机。

### 为什么不直接用纯 FSDP

FSDP 更适合固定 forward-backward 训练流程，但 Agent RL 训练存在以下动态特征：

- 动态生成导致序列长度变化大
- 工具调用会引入分支
- 需要异步推理与训练协同

因此项目采用 Ray 负责编排，vLLM 负责生成，FSDP 负责训练。

### 拆分原则

| 模块 | 拆分原则 | 核心类/函数 | 输入 | 输出 |
| --- | --- | --- | --- | --- |
| Rollout | 生成与训练分离 | `actor_rollout_wg.generate_sequences()` | prompt | responses、logits |
| Reward | 评分独立 | `compute_reward()` | batch | reward_tensor |
| Actor Update | 策略更新集中管理 | `compute_policy_loss_entropy_balanced_clipping()` | old_log_prob、entropy、advantages | pg_loss |

### 模块间通信方式

| 通信方式 | 适用场景 | 本项目选择 |
| --- | --- | --- |
| Ray RPC | Worker 间通信 | Actor、Critic、RefPolicy 调用 |
| TensorDict | 批量 tensor 数据传递 | `DataProto.batch` |
| 共享内存 | 高速数据交换 | 当前不使用 |
| Gloo / NCCL | 梯度同步 | FSDP 内部使用 |

## ARAEPO 算法细节

### 熵感知 Advantage

标准 GRPO 对所有 token 使用相同权重：

```python
advantage = (reward - mean) / std
```

ARAEPO 会根据 token entropy 调整 advantage：

```python
advantage = advantage * (1 + 0.2 * entropy_normalized)
```

量化效果示例：

| token 类型 | `entropy_normalized` | advantage 变化 |
| --- | ---: | --- |
| 高不确定 token | `1.0` | 放大 20% |
| 低不确定 token | `-1.0` | 缩小 20% |

### 熵处理方案对比

| 方案 | 公式 | 优势 | 劣势 |
| --- | --- | --- | --- |
| 直接使用熵 | `advantage * entropy` | 简单 | 不同模型之间不可比 |
| 熵排名 | `advantage * rank(entropy)` | 鲁棒 | 丢失幅度信息 |
| 标准化熵 | `advantage * (1 + 0.2 * (entropy - mean) / std)` | 可比、通用 | 多一个超参数 |

本项目选择标准化熵方案。

### 熵标准化错误处理

```python
# core_algos.py
if len(valid_entropy_flat) > 0:
    entropy_mean = valid_entropy_flat.mean()
    entropy_std = valid_entropy_flat.std()
    if entropy_std == 0:
        entropy_std = 1.0
    entropy_normalized = (entropy - entropy_mean) / entropy_std
    advantages = advantages * (1 + 0.2 * entropy_normalized.detach())
```

需要处理的异常场景：

1. 组内只有一个样本，无法稳定计算标准差。
2. 所有 token 熵相同，`std = 0`，标准化需要退化处理。

### 自适应裁剪机制

标准 PPO 裁剪范围固定：

```text
[1 - eps, 1 + eps] = [0.8, 1.2]
```

ARAEPO 使用自适应上界：

```python
max_bound = (1 + cliprange_high) / ratio.detach() * ratio
```

当 `ratio > 1`，策略概率增加：

```text
假设 ratio = 1.5, ratio_old = 1.0
max_bound = 1.2 / 1.0 * 1.5 = 1.8
clamp(1.5, 0.8, 1.8) = 1.5
```

当 `ratio < 1`，策略概率减少：

```text
假设 ratio = 0.5, ratio_old = 1.0
max_bound = 1.2 / 1.0 * 0.5 = 0.6
clamp(0.5, 0.8, 0.6) = 0.6
```

### 裁剪阶段策略

| 阶段 | `cliprange_low` | `cliprange_high` | 说明 |
| --- | ---: | ---: | --- |
| 初期，0-20% | `0.2` | `0.3` | 宽松裁剪，鼓励探索 |
| 中期，20-80% | `0.2` | `0.2` | 标准裁剪 |
| 后期，80-100% | `0.2` | `0.15` | 收紧裁剪，稳定收敛 |

### 防止异常的关键代码

```python
# core_algos.py
if enable_entropy_balanced_clipping:
    min_bound = torch.full_like(ratio, 1 - cliprange_low)
    max_bound = (1 + cliprange_high) / ratio.detach() * ratio
    pg_losses2 = -advantages * torch.clamp(ratio, min_bound, max_bound)
else:
    pg_losses2 = -advantages * torch.clamp(
        ratio,
        1 - cliprange_low,
        1 + cliprange_high,
    )
```

`ratio.detach()` 会断开梯度，避免裁剪边界本身引入不稳定梯度。

### 终止条件

| 条件 | 说明 |
| --- | --- |
| `total_training_steps` 达到 | 正常终止 |
| `global_steps >= self.total_training_steps` | 训练循环退出 |
| `early_stop` 信号 | 可选；验证指标长期停滞时触发 |

## 扩展与替换实现

### 新增 ValueNorm 模块

步骤 1：在 `core_algos.py` 中添加函数。

```python
# core_algos.py
def compute_value_norm(advantages, response_mask, beta=0.99, eps=1e-8):
    """Normalize advantages before policy update."""
    ret = advantages
    ret = ret / (torch.std(ret) + eps)
    return ret
```

步骤 2：在 `ray_trainer.py` 的 `fit()` 中集成。

```python
# ray_trainer.py
batch = compute_advantage(batch, adv_estimator=...)

if self.config.algorithm.get("use_value_norm", False):
    batch.batch["advantages"] = compute_value_norm(
        batch.batch["advantages"],
        batch.batch["response_mask"],
    )
```

步骤 3：在 YAML 配置中添加开关。

```yaml
algorithm:
  use_value_norm: true
  adv_estimator: grpo
```

### 将 GRPO 替换为 GAE

配置层修改：

```yaml
algorithm:
  adv_estimator: gae
  gamma: 0.99
  lam: 0.95
```

代码层通常无需修改，`compute_advantage()` 会根据 `adv_estimator` 自动选择实现：

```python
# ray_trainer.py
if adv_estimator == AdvantageEstimator.GAE:
    advantages, returns = core_algos.compute_gae_advantage_return(...)
elif adv_estimator == AdvantageEstimator.GRPO:
    advantages, returns = core_algos.compute_grpo_outcome_advantage(...)
```
