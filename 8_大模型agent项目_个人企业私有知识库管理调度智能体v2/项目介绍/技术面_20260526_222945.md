# com_agent_chat — 技术面总结

## 项目基础问题

### Q1. 介绍一下这个项目？

**参考答案：** 这是一个基于 4-Agent 混合编排的企业知识管理系统。核心功能是：上传文档后自动解析和构建知识图谱，用户用自然语言提问，系统同时做向量语义搜索和知识图谱多跳推理，融合后生成带来源引用的答案，文档更新后知识库自动增量同步。我独立完成了架构设计、4 个 Agent 的编码实现、GraphRAG 混合检索管道、前后端和部署配置。

**面试官想听的点：** 项目类型定位准确、核心价值一句话讲清、个人角色明确、技术栈匹配

**常见踩坑：** 只说"这是一个 RAG 项目"，不讲个人角色；或罗列一堆技术名词但没有说明它们之间的关系。

---

### Q2. 这个项目和同类项目有什么区别？

**参考答案：** 大多数 RAG 项目只做向量检索，本项目同时维护了向量库和知识图谱两套知识表示，让问答可以融合语义和结构化两种信息源。另外多数项目只关注"怎么查"，本项目从文档入库、知识抽取、混合检索到增量更新覆盖了知识的全生命周期。

**面试官想听的点：** 差异化设计、具体的对比维度、解决了什么别人没解决的问题

**常见踩坑：** 笼统地说"更好"，没有说出具体好在哪。

---

## 核心技术问题

### 项目整体设计（6题）

### Q3. 为什么选择 LangGraph 做 Agent 编排？

**参考答案：** 对比了 LangGraph、CrewAI 和 AutoGen。LangGraph 用有向图 StateGraph 定义流程，控制粒度最细——我可以精确控制每个节点的输入输出和条件路由（比如更新流水线中失败自动 retry）。CrewAI 上手快但基于角色定义，流程控制不够精细。AutoGen 偏研究。本项目有 3 条独立的流水线和条件路由，LangGraph 最合适。

**面试官想听的点：** 框架对比经历、选型理由、根据场景选择工具的能力

**常见踩坑：** 只说"LangGraph 是主流"没有对比过程。

---

### Q4. 你是怎么划分 4 个 Agent 的职责的？

**参考答案：** 按知识处理的生命周期划分：文档解析负责"看懂"文档（多模态→文本）、知识抽取负责"提炼"信息（文本→三元组）、问答负责"使用"知识（问题→检索→答案）、知识更新负责"维护"知识（变更→同步）。每个 Agent 的 Prompt 只覆盖自己的职责范围，避免单 Agent Prompt 膨胀导致的指令冲突。

**面试官想听的点：** 职责单一原则的理解、Agent 粒度的把握、Prompt 工程意识

**常见踩坑：** 说不清为什么是 4 个而不是 3 个或 5 个。

---

### Q5. 三条流水线之间的关系是什么？

**参考答案：** 三条流水线在 LangGraph 中是独立的 StateGraph 实例，不共享状态。入库流是最基础的——先有知识才能问答；问答流是最频繁调用的——每次用户提问走一次；更新流是对入库流的增量补充——文档变了不走完整入库流，走更新流。它们在 API 层被分别调用：上传文件触发入库流，提问触发问答流，手动/自动触发更新流。

**面试官想听的点：** 系统全局视角、流水线间的协作关系

**常见踩坑：** 只说三条流水线的名字，不说它们之间的关系。

---

### Q6. 系统的状态管理是怎么做的？

**参考答案：** LangGraph 的 StateGraph 内置状态传递机制。每个节点函数接收 state dict，返回更新后的 state dict。入库流的状态是 `{file_paths, chunks, extractions, vectors_stored, entities_stored}`，节点之间通过 state 传递数据。LangGraph 还支持 checkpointer 做状态持久化和断点续传，但本项目当前没有启用，因为三条流水线都是短生命周期的一次性执行。

**面试官想听的点：** 对 LangGraph 状态机制的理解、为什么没有用 checkpointer

**常见踩坑：** 混淆 LangGraph state 和数据库 state。

---

### Q7. 为什么不用微调而是用 RAG？

**参考答案：** 这个场景的核心需求是"基于企业最新文档回答"，知识频繁更新——今天上传了新的产品手册，明天的问答就要基于新内容。RAG 改变的是推理时的上下文，文档更新后知识库跟着更新，答案自动反映最新信息。微调改变的是模型权重，每次文档更新都要重新训练，对于动态知识场景不划算。而且 RAG 的每个答案可以附带来源引用，方便人工核实，微调的答案来源不可追溯。

**面试官想听的点：** RAG vs 微调的适用场景判断、可解释性意识

**常见踩坑：** 说"微调太贵所以选 RAG"——成本不是关键差异，适用场景才是。

---

### Q8. 这个系统的模块间通信方式是什么？

**参考答案：** Agent 之间不直接通信——它们通过 LangGraph 的 StateGraph 编排器间接协作。编排器负责把上一个节点的输出作为下一个节点的输入传入 state。Agent 与服务层（VectorStoreService、KnowledgeGraphService）之间是通过构造函数注入依赖的，Agent 持有服务引用直接调用方法。这种设计让 Agent 之间解耦——修改 QAAgent 不影响其他 Agent。

**面试官想听的点：** 解耦设计、依赖注入、编排器模式的理解

**常见踩坑：** 说"Agent 之间通过消息队列通信"——不符合本项目实际设计。

---

### 核心技术/RAG（6题）

### Q9. 什么是 RAG？为什么它比直接让模型回答更适合企业知识问答？

**参考答案：** RAG 就是在 LLM 生成答案之前，先从外部知识库检索相关文档作为上下文拼入 Prompt，让模型基于真实数据回答。相比直接让模型回答，RAG 的答案基于实际检索到的内容而非模型记忆，不会产生幻觉；企业知识随时更新，RAG 能及时反映最新信息；每个答案可以附带来源引用，方便核实。

**面试官想听的点：** RAG 的核心价值（反幻觉、实时性、可追溯）

**常见踩坑：** 把 RAG 说成是微调的替代品——它们是互补关系，不是替代关系。

---

### Q10. 为什么说 Chunking 决定了检索系统的上限？

**参考答案：** Chunk 太小会丢失上下文——"他负责微信产品线"如果和"张三担任腾讯CEO"不在同一个 Chunk，检索"张三负责什么"时可能只命中后半段，无法确定"他"就是张三。Chunk 太大引入过多无关内容，增加 LLM 推理噪音。本项目选 512 字符 + 64 重叠——512 字符能覆盖大多数完整段落，64 重叠保证 Chunk 边界处的信息不被截断。

**面试官想听的点：** 对 Chunk 大小 trade-off 的理解、有具体数字和经验支撑

**常见踩坑：** 只说"512 是推荐值"没有解释为什么。

---

### Q11. Embedding 模型怎么选？本项目用了什么？

**参考答案：** 本项目用 `shibing624/text2vec-base-chinese`，384 维，本地 CPU 运行。选它的原因是中文语义理解好、开源免费、轻量（本地 CPU 推理不占用 GPU/API 额度）。对比过几个选项：OpenAI text-embedding-3-small（质量好但要 API 费用和网络延迟）、bge-large-zh（1024 维，质量最佳但模型更大推理更慢）。384 维的 text2vec 在这个场景下性价比最高。

**面试官想听的点：** Embedding 选型的多维度考量（质量/成本/延迟/维度）

**常见踩坑：** 只列模型名字不解释选择理由。

---

### Q12. 为什么很多系统要加 Rerank？

**参考答案：** 向量检索用近似最近邻（ANN）追求速度，精度有损。第一次检索可能将语义相关但不是最佳答案的文档排在前面。Rerank（重排序）用更精确的模型对 Top-K 做二次评分——计算量更大但只算 K 个候选，不增加多少延迟。本项目用交叉重排序（cross rerank），4 种检索来源的结果放在一起统一排序，不同来源设不同权重（路径推理 1.25 > 子图 1.15 > 向量 1.0）。

**面试官想听的点：** 理解向量检索的精度损失、二阶段检索的设计思路

**常见踩坑：** 混淆重排序（rerank）和混合检索（hybrid retrieval）——前者是排序阶段，后者是检索阶段。

---

### Q13. 向量数据库怎么选型？

**参考答案：** 本项目默认用 ChromaDB（嵌入式，零配置，适合原型和中小规模），同时支持切换到 PGVector（适合已有 PostgreSQL 的企业）。对比过几个选项：Milvus 性能最佳支持百亿级但运维复杂，Pinecone 全托管零运维但贵且数据不可控，Weaviate 支持混合搜索但社区小。当前阶段选 ChromaDB 是因为它完全嵌入式不需要额外服务，适合本地开发和演示。生产环境会迁到 PGVector 或 Milvus。

**面试官想听的点：** 多维度对比（规模/运维/成本）、不同阶段的选型策略

**常见踩坑：** 说"ChromaDB 是最好的向量数据库"——没有最好的，只有最适合场景的。

---

### Q14. Hybrid Retrieval 的本质是什么？

**参考答案：** 同时使用多种检索方法取各自 Top 结果后加权融合。不同方法的优势互补——向量检索擅长语义理解（"水果"和"苹果"），关键词检索擅长精确匹配（"订单号#2025-001"），知识图谱擅长关系推理（"张三→所属部门→技术部→部门负责人→李四"）。本项目是向量检索 + 图谱检索的双路混合，图谱路径推理给 1.25 倍最高权重因为它直接提供推理链，是回答多跳问题的最有价值信息。

**面试官想听的点：** 理解不同检索方法的互补性、有权重设计的思考

**常见踩坑：** 只说"混合检索更准"没有解释为什么不同方法能互补。

---

### 核心技术/Agent（6题）

### Q15. 什么是 Agent？跟传统的 Chain 有什么区别？

**参考答案：** Agent 是能自主决策的 AI 程序，包含 LLM 核心、规划能力、记忆和工具使用。Chain 执行预定义的固定流程——先 A 后 B 再 C，不管中间结果如何都走固定路径。Agent 根据中间结果动态决定下一步——如果检索结果置信度低，可以改写 Query 重新检索。类比：Chain 是流水线工人，Agent 是项目经理。

**面试官想听的点：** 自主决策 vs 固定流程的核心差异

**常见踩坑：** 把"调了 LLM 的就是 Agent"——Chain 也调 LLM，区别在于是否有动态决策。

---

### Q16. 多 Agent 系统的编排模式有哪些？

**参考答案：** 三种核心模式。主从模式（Boss-Worker）：中央协调者分配任务给 Worker，适合有明确分工的场景——本项目接近这种模式，LangGraph 编排器充当 Boss，4 个 Agent 是 Worker。流水线模式（Pipeline）：上游输出是下游输入，适合线性流程——入库流水线就是纯 Pipeline。讨论模式（Joint Discussion）：多 Agent 平等对话协作，适合需要多视角分析的场景，但容易陷入无限循环。

**面试官想听的点：** 三种模式的区别和适用场景、本项目用了哪种

**常见踩坑：** 把本项目的更新流（有条件路由）说成是讨论模式。

---

### Q17. 如何解决多 Agent 间的无限循环？

**参考答案：** 本项目用了两招。第一，LangGraph 的 StateGraph 是显式有向图，循环必须显式加边——不像对话式 Agent 容易在对话中来回。第二，更新流中的条件路由有明确的终止条件：`should_continue` 判断是否还有失败的更新，没有就走向 END。通用的防循环手段还包括：最大迭代次数、Token 预算（超预算强制终止）、消息摘要（历史过长时压缩）。

**面试官想听的点：** 有向图的天然优势、终止条件的设计

**常见踩坑：** 只说"LangGraph 不会死循环"——有向图只是降低了风险，条件路由如果设计不好仍然可能来回跳。

---

### Q18. 你的 Agent 系统如何保证容错性？

**参考答案：** 分层容错。Agent 层：每个处理函数 try-except 包裹，异常不阻断下游——入库流中 PDF 解析失败只影响该文件的 chunks 为空，不影响 HTTP 响应返回。服务层：向量库不可用时跳过向量存储只写图谱，图谱不可用时同理。编排层：更新流中失败的任务自动进入 retry 节点重试。环境层：Embedding 模型子进程隔离，即使模型崩溃不影响主进程。

**面试官想听的点：** 分层容错的设计思路、具体措施而非笼统说"有容错"

**常见踩坑：** 只说"有 try-except"——这是最基本的，要说出分层的设计。

---

### Q19. 你的系统如何处理幻觉问题？

**参考答案：** 三层防护。第一层是检索增强——答案必须基于检索到的上下文生成，Prompt 明确要求"答案必须基于提供的上下文，不要编造"。第二层是来源引用——每个答案标注 `[来源: xxx]`，用户可以直接核实。第三层是空库兜底——知识库为空时明确告知用户"当前知识库为空"而非强行编造。图谱也在结构层面约束——Neo4j 中存储的结构化事实（如"张三 WORKS_AT 腾讯"）是不可篡改的硬事实。

**面试官想听的点：** 多层防护、可追溯、不强行编造的原则

**常见踩坑：** 说"RAG 可以完全消除幻觉"——RAG 减少幻觉但不能消除，检索到的上下文本身可能包含错误信息。

---

### Q20. 为什么用 4 个 Agent 而不是 1 个全能的 Agent？

**参考答案：** 单 Agent 的问题：Prompt 膨胀——要覆盖文档解析、知识抽取、问答、更新四种场景，系统提示词冲突严重（解析要求"尽可能完整"，问答要求"简洁精准"）；工具过载——挂载太多工具后 LLM 选择工具的准确率下降；难以调试——出问题不知道是解析环节还是抽取环节。4 个 Agent 各自 Prompt 精简聚焦，独立迭代不影响全局，故障隔离。

**面试官想听的点：** 单 Agent 的工程痛点、职责单一原则

**常见踩坑：** 只说"多 Agent 更高级"——要说出单 Agent 的具体问题。

---

### 核心技术/GraphRAG（4题）

### Q21. 知识图谱在 RAG 中起什么作用？

**参考答案：** 补充向量检索的两个盲区。一是结构化关系——"张三的老板是谁"这个问题，向量检索只能找到包含"张三"和"老板"的文档，但不理解"张三→部门→主管"这个关系链。图谱直接存储这个关系链，Cypher `MATCH path = shortestPath(...)` 可以一步到位。二是多跳推理——"张三负责的产品的竞争对手"需要 3 跳，向量检索几乎无法处理。

**面试官想听的点：** 向量检索的局限、图谱的互补价值

**常见踩坑：** 说"知识图谱比向量检索更好"——不是更好，是互补。

---

### Q22. GraphRAG 的交叉重排序权重是怎么确定的？

**参考答案：** 权重的设计逻辑是：路径推理链（1.25）直接回答多跳问题，信息价值最高→子图遍历（1.15）提供结构化关系，比纯语义相似更精准→社区摘要（1.1）提供高层概览，补充细节→向量检索（1.0）是基线，覆盖面最广但精度相对低。这些权重是经验值，不是通过网格搜索调出来的——实际使用中可以根据具体数据集的检索效果微调。代码在 `services/graph_rag.py:220`。

**面试官想听的点：** 权重有逻辑支撑、知道是经验值、提到了可调

**常见踩坑：** 说"1.25 是通过 AB 测试确定的"——本项目没有做 AB 测试，诚实说经验值更好。

---

### Q23. Neo4j 的子图检索是怎么做的？

**参考答案：** 用可变长度路径匹配 `MATCH path = (start:Entity {name: $name})-[*1..2]-(neighbor)`，从查询中识别到的实体出发，遍历 1-2 跳内的所有邻居节点和关系。限制在 2 跳是因为超过 2 跳的关系通常太远，信息相关度大幅下降，而且 Cypher 的无限制路径遍历在大图上性能很差。加上 LIMIT 50 防止子图过大。

**面试官想听的点：** Cypher 可变长度路径的理解、为什么限制 hop 数

**常见踩坑：** 只说"查 Neo4j"不说具体 Cypher 怎么写。

---

### Q24. 路径检索和子图检索有什么区别？

**参考答案：** 子图检索是做广度扩散——从实体出发找所有 2 跳内的邻居，回答"和这个实体相关的有哪些"。路径检索是做定向连接——找两个实体之间的最短路径，回答"这两个实体通过什么关系链连接"。路径检索用了 `shortestPath()` 函数和 `LIMIT 3`，避免全图搜索。路径推理链是回答"X 和 Y 有什么关系"这类问题的直接证据，所以给了最高权重 1.25。

**面试官想听的点：** 两种检索策略的差异和各自适用的问题类型

**常见踩坑：** 把子图检索和路径检索混为一谈。

---

### CDC 增量更新（4题）

### Q25. CDC 的核心思想是什么？相比全量更新优势在哪？

**参考答案：** CDC 的核心是"监听变更事件，只处理变化的部分"，而不是定期全量扫描。全量更新 1000 个文件每次都要重新解析所有文件，即使只改了 5 个也要跑 30 分钟。CDC 只处理变化的 5 个文件，大约 30 秒。本项目支持两种 CDC 源：文件系统 Watchdog（监听本地文件创建/修改/删除）和 Kafka（消费数据库 CDC 事件，支持 Debezium 格式）。

**面试官想听的点：** 增量 vs 全量的效率差异、两种 CDC 源的区别

**常见踩坑：** 说"CDC 就是 Kafka"——Kafka 是 CDC 的一种实现方式，文件系统监听也是 CDC。

---

### Q26. 如何区分大改和小改？处理策略有什么不同？

**参考答案：** 用 `compute_diff` 计算文本变化率。对比新旧文本的行集合差异：新增行+删除行占总行数的比例。超过 30% 视为大改（`is_major_change=True`），理论上应删除旧数据重新全量解析；低于 30% 视为小改，只更新变化的 Chunk。30% 的阈值是经验值——低于这个值的改动通常是局部修改（改了某一段），重做局部即可；超过这个值说明文档结构发生了重大变化，全量重建更可靠。

**面试官想听的点：** 差量计算的具体方法、阈值的含义

**常见踩坑：** 只说"有大改和小改"不说怎么判定。

---

### Q27. 版本号管理有什么用？

**参考答案：** 每次实体更新时版本号+1（`bump_version`），记录在 Neo4j 节点的 `version` 属性中。用途：追踪一个实体被修改过几次（高频更新的实体可能是热点），支持回滚到历史版本（虽然当前只存了版本号没存历史快照），增量更新的去重依据（版本号变化才说明真的更新了）。版本号存在内存 dict 中，进程重启会丢失——生产环境应该持久化到数据库。

**面试官想听的点：** 版本管理的实际用途、对局限性的认知

**常见踩坑：** 说"版本管理可以回滚到任意历史版本"——当前实现只存了版本号，没存快照。

---

### Q28. 如果 Kafka 不可用，增量更新还能工作吗？

**参考答案：** 可以。增量更新有两条独立路径——Watchdog 文件监听和 Kafka 消息消费。Watchdog 是文件系统级别的实时监听（基于 inotify/ReadDirectoryChanges），不依赖任何外部服务。Kafka 是数据库级别的 CDC（如 MySQL binlog → Debezium → Kafka），用于监听数据库中的知识变更。两者互为补充，任意一个不可用都不影响另一个。当前默认只启用了 Watchdog。

**面试官想听的点：** 双模式的独立性、降级策略

**常见踩坑：** 说"没有 Kafka 增量更新就废了"。

### 工程化（4题）

### Q29. FastAPI 的 Lifespan 是怎么管理服务生命周期的？

**参考答案：** 在 `lifespan` 异步上下文管理器中，应用启动时依次初始化：创建 uploads 目录 → 初始化向量库连接 → 初始化 Neo4j 连接 → 构建三条 LangGraph 流水线并注入依赖。应用关闭时（yield 之后）关闭 Neo4j 连接。如果某个服务初始化失败（如 Neo4j 没启动），try-except 捕获异常不阻断应用启动——核心问答功能不可用但 API 仍然可以响应健康检查。

**面试官想听的点：** 启动顺序设计、优雅降级、asynccontextmanager 的使用

**常见踩坑：** 只说"用了 FastAPI"不解释生命周期管理。

### Q30. Embedding 模型为什么要放在子进程？

**参考答案：** sentence-transformers 底层依赖 PyTorch，PyTorch 的 C 扩展和 Python asyncio 事件循环存在的线程安全问题会导致 segfault。将模型加载到独立子进程中，主进程通过 multiprocessing.Queue 发送编码请求、接收向量结果，从物理上隔离了 C 扩展和事件循环。这个方案比"全部改为同步调用"或"换用 API Embedding"更平衡——保留了异步框架的优势，不增加外部依赖。

**面试官想听的点：** 问题的根因分析、方案对比、工程权衡

**常见踩坑：** 只说"为了性能"——不是为了性能，是为了稳定性。

---

### Q31. API 密钥怎么管理？有什么安全措施？

**参考答案：** 所有密钥通过环境变量注入（.env 文件），由 pydantic-settings 自动加载，不硬编码。.env 加入了 .gitignore 不会被提交到版本控制。代码中的 Settings 类有默认值（如 `openai_api_key: str = ""`），没有密钥不会泄露。文档中所有密钥统一替换为 `<敏感信息>` 占位符。需要加强的地方：当前没有密钥加密存储、没有 API 鉴权（任何人都可以调接口）、Cypher 查询虽然是参数化的但 LLM 生成的 Cypher 没有安全审查。

**面试官想听的点：** 安全意识和已知不足都坦诚说出来

**常见踩坑：** 说"我们的系统非常安全"——坦诚说不足更显专业。

### Q32. 如果知识库有 100 万文档，系统怎么扩展？

**参考答案：** 分几个层面。向量库：从 ChromaDB 嵌入式模式迁移到 Milvus 分布式集群（支持百亿级向量+分区索引）。文档解析：多 Worker 异步并行，Celery 任务队列，解析和抽取解耦。图谱：Neo4j 集群+读写分离，对大实体做分片。API 层：水平扩展（Kubernetes 多副本+Load Balancer）。CDC：Kafka 多分区提升吞吐。加上 Embedding 缓存（相同文本不重复编码）和语义缓存（相同问题不重复检索）。

**面试官想听的点：** 各层扩展方案的可行性和具体技术选型

**常见踩坑：** 只说"加机器"没有各层的具体方案。

### 系统设计（3题）

### Q33. 设计模式在项目中的具体应用？

**参考答案：** 策略模式——DocParserAgent 根据文件扩展名路由到不同的解析方法。管道模式——入库流水线的 parse→extract→store 串行链。单例模式——Settings 全局实例和 EmbeddingClient 单例。代理模式——EmbeddingClient 作为子进程 Embedding 模型的代理。模板方法——知识抽取定义抽取骨架（LLM 调用→JSON 解析→构造对象），解析细节可变。

**面试官想听的点：** 设计模式的具体应用而非纸上谈兵

**常见踩坑：** 说"用了 MVC 模式"——本项目没分 View 层。

---

### Q34. 系统的性能瓶颈在哪里？怎么定位和优化？

**参考答案：** 最大瓶颈是 LLM 调用延迟——知识抽取每 Chunk 一次 LLM 调用，处理 45 个 Chunk 的 PDF 需要 45 次 API 往返。优化方向：批量 Prompt（一次调用处理 5-10 个 Chunk）。次瓶颈是 Embedding 推理——384 维的 text2vec 在 CPU 上推理，45 个 Chunk 大约需要几秒。优化方向：GPU 推理或切换到 API Embedding。最小的瓶颈是 Cypher 查询——当前数据量小，Neo4j 单机完全够用。

**面试官想听的点：** 能区分主要矛盾和次要矛盾、有量化意识

**常见踩坑：** 说"ChromaDB 是瓶颈"——在几千文档的规模下，真正的瓶颈是 LLM 调用。

---

### Q35. 这个系统的代码组织有什么特点？

**参考答案：** 按功能分层而非按文件类型：agents/（业务逻辑）、services/（数据访问）、orchestrator/（流程编排）、api/（接口适配）、config/（配置管理）。依赖方向是单向的——api→orchestrator→agents→services→config，没有循环依赖。Agent 通过构造函数注入依赖（如 QAAgent 接收 vector_store 和 knowledge_graph），而不是内部创建，方便测试和替换实现。

**面试官想听的点：** 分层架构、依赖方向、依赖注入

**常见踩坑：** 只说"代码按文件夹分了"没有解释分层的原因。

### Q38. 测完接口性能后，你思考过如何优化吗？

**参考答案：** 做了几个优化方向的推演。语义缓存：相同或相似的问题直接返回缓存答案，预期命中率 15-20%（企业内部 FAQ 类问题重复率较高）。检索预热：启动时对高频实体做预计算（子图邻接表、Embedding 缓存），首次查询延迟降 30%。连接池：当前 Neo4j 每次查询创建新 session，改为连接池复用。批量 Embedding：45 个 Chunk 一次编码而不是逐个编码。具体数字需要压测后才能确定，但这些方向都有依据。

**面试官想听的点：** 性能分析的方法论、优化方向有逻辑支撑、不虚报数字

**常见踩坑：** 说"加缓存就快十倍"——没有分析缓存命中率就报数字。
