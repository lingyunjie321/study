## 2. 数据模型 (DocParserAgent 数据部分)

**文件**: `code/python/agents/doc_parser_agent.py`

在深入业务逻辑之前，先看数据模型定义——它们是整个系统各模块之间的"通用语言"。

```python
class DocType(str, Enum):
    """文档类型枚举 — 决定走哪个解析器"""
    PDF = "pdf"
    IMAGE = "image"
    TABLE = "table"
    TEXT = "text"
    MARKDOWN = "markdown"
    UNKNOWN = "unknown"

@dataclass
class DocumentChunk:
    """文档块 — 向量检索的基本单元"""
    content: str                                     # 文本内容
    doc_id: str                                      # SHA256前16位，关联源文档
    chunk_index: int                                 # 在文档中的序号(0,1,2...)
    doc_type: DocType                                # 来源文档类型
    metadata: dict[str, Any] = field(default_factory=dict)  # {source, char_start, char_end}
    embedding: list[float] | None = None             # 384维向量，延迟计算
```

**设计要点**: `doc_id` 用 SHA256 前 16 位而不是完整哈希——16 位足够在中小规模文档集中保证唯一性，且短 ID 在日志和调试中更友好。

**面试考点**: "为什么用 dataclass 而不是 Pydantic Model？"——dataclass 更轻量，没有验证开销（数据已经在代码中构造，不需要运行时校验）。

## 3. 文档解析 (agents/doc_parser_agent.py)

**文件**: `code/python/agents/doc_parser_agent.py`

文档解析 Agent 是多模态文档的统一入口。核心设计是"自动分类→路由解析→统一分块"。

```python
class DocParserAgent:
    # 扩展名 → 文档类型映射表
    SUPPORTED_EXTENSIONS: dict[str, DocType] = {
        ".pdf": DocType.PDF, ".png": DocType.IMAGE,
        ".jpg": DocType.IMAGE, ".csv": DocType.TABLE,
        ".xlsx": DocType.TABLE, ".txt": DocType.TEXT, ".md": DocType.MARKDOWN,
    }
    CHUNK_SIZE = 512       # 每个chunk的字符数
    CHUNK_OVERLAP = 64     # 相邻chunk的重叠字符数

    async def parse(self, file_path: str) -> list[DocumentChunk]:
        """分类→解析→分块，三步合一"""
        doc_type = self._classify(file_path)        # ① 按扩展名分类
        doc_id = self._make_doc_id(file_path)        # ② 生成文档ID
        raw_texts = []                                # ③ 路由到对应解析器
        if doc_type == DocType.PDF:
            raw_texts = await self._parse_pdf(file_path)
        elif doc_type == DocType.IMAGE:
            raw_texts = await self._parse_image(file_path)
        elif doc_type == DocType.TABLE:
            raw_texts = await self._parse_table(file_path)
        elif doc_type in (DocType.TEXT, DocType.MARKDOWN):
            raw_texts = self._parse_text(file_path)
        chunks = self._chunk_texts(raw_texts, doc_id, doc_type, file_path)  # ④ 分块
        return chunks
```

**PDF 解析的分层降级**（最核心的设计）:

```python
async def _parse_pdf(self, file_path: str) -> list[str]:
    texts: list[str] = []
    try:
        from PyPDF2 import PdfReader
        reader = PdfReader(file_path)
        for page in reader.pages:
            page_text = page.extract_text() or ""
            if page_text.strip():
                texts.append(page_text.strip())
    except Exception:
        texts.append(f"[PDF 解析失败] {file_path}")

    if not texts:  # ← 关键降级点：文字提取为空 → LLM视觉兜底
        texts = await self._pdf_vision_fallback(file_path)
    return texts
```

**设计要点**: PDF 文字提取失败不报错，而是静默切换到 LLM 视觉方案——用户无感知。

**面试考点**: "PDF 解析为什么不用 PyMuPDF (fitz)？"——PyMuPDF 需要额外安装系统依赖，PyPDF2 是纯 Python 安装最简单。对于复杂 PDF 可以替换为 PyMuPDF。

## 4. 知识抽取 (agents/knowledge_extract_agent.py)

**文件**: `code/python/agents/knowledge_extract_agent.py`

将非结构化的文档文本转为结构化的知识三元组。核心：LLM + JSON Schema Prompt + 跨 Chunk 去重。

```python
class KnowledgeExtractAgent:
    BATCH_SIZE = 5  # 每批处理5个chunk

    async def extract(self, chunks: list[DocumentChunk]) -> list[ExtractionResult]:
        results = []
        for i in range(0, len(chunks), self.BATCH_SIZE):     # 按批处理
            batch = chunks[i : i + self.BATCH_SIZE]
            for chunk in batch:
                result = await self._extract_from_chunk(chunk)  # LLM抽取
                results.append(result)
        return self._deduplicate(results)                    # 跨chunk去重
```

**去重逻辑**——避免同一个"张三"在 45 个 Chunk 中出现时建 45 个节点：

```python
@staticmethod
def _deduplicate(results: list[ExtractionResult]) -> list[ExtractionResult]:
    seen_entities: dict[str, Entity] = {}                    # 实体去重 Map
    seen_relations: set[tuple[str, str, str]] = set()        # 关系去重 Set

    for result in results:
        for ent in result.entities:
            key = f"{ent.name}::{ent.type}"                  # 同名+同类型 = 同一实体
            if key not in seen_entities:
                seen_entities[key] = ent

        for rel in result.relations:
            key = (rel.head, rel.relation, rel.tail)         # 完全相同的三元组只保留一个
            if key not in seen_relations:
                seen_relations.add(key)
```

**设计要点**: 去重用 `dict` 和 `set` 而非数据库去重——内存操作比 CQL 查询快 1000 倍。

**面试考点**: "如果两个不同的人同名怎么办？"——当前设计会合并为同一个实体。解决方案是引入实体消歧（Entity Linking），利用上下文信息（如"张三，腾讯CEO"vs"张三，阿里工程师"）区分同名实体。这是知识图谱领域的经典难点。

**ChromaDB 安全措施**：所有 ChromaDB 的 C API 调用通过线程池执行，避免 asyncio segfault：

```python
async def _run_sync(self, fn, *args, **kwargs):
    """将 ChromaDB C 调用放到线程池，避免阻塞事件循环"""
    loop = asyncio.get_event_loop()
    return await loop.run_in_executor(self._executor, lambda: fn(*args, **kwargs))
```

**设计要点**: 手动维护向量计数（`_stored_count`）而非调用 ChromaDB API `collection.count()`——避开有 segfault 风险的 C 扩展调用。

**面试考点**: "ChromaDB 和 Milvus 的区别？"——ChromaDB 是嵌入式向量库，零配置适合原型；Milvus 是分布式向量库，支持百亿级向量适合生产。本项目的 `VectorStoreService` 抽象层使得迁移只需改 `_init` 方法。

## 6. 知识图谱 (services/knowledge_graph.py)

**文件**: `code/python/services/knowledge_graph.py`

Neo4j 操作封装，核心是用 MERGE 语义实现幂等的实体和关系写入。

```python
class KnowledgeGraphService:
    async def init(self):
        from neo4j import AsyncGraphDatabase
        self._driver = AsyncGraphDatabase.driver(       # 异步驱动
            settings.neo4j_uri,
            auth=(settings.neo4j_user, settings.neo4j_password),
        )
        await self._ensure_indexes()                     # 创建 name/type/source 索引

    async def upsert_entity(self, entity, version=1, source=""):
        """创建或更新实体 — MERGE 保证幂等"""
        cypher = """
        MERGE (e:Entity {name: $name})            -- 以name为唯一键
        ON CREATE SET                             -- 新实体：初始化全部字段
            e.type = $type, e.description = $description,
            e.version = $version, e.source = $source,
            e.created_at = $now, e.updated_at = $now
        ON MATCH SET                              -- 已有实体：只更新描述和版本
            e.description = CASE WHEN $description <> '' THEN $description ELSE e.description END,
            e.version = $version, e.updated_at = $now
        """
        async with self._driver.session() as session:
            await session.run(cypher, params)

    async def get_neighbors(self, entity_name, hops=2):
        """多跳子图遍历 — GraphRAG 核心"""
        cypher = f"""
        MATCH path = (start:Entity {{name: $name}})-[*1..{hops}]-(neighbor)
        RETURN start.name AS source,
               [r IN relationships(path) | type(r)] AS relations,
               neighbor.name AS target, neighbor.type AS target_type,
               neighbor.description AS target_desc
        LIMIT 50
        """
        return await self.execute_cypher(cypher, {"name": entity_name})
```

**设计要点**: `ON CREATE` 和 `ON MATCH` 处理不同逻辑——新建时初始化全部字段，已存在时只更新描述（保留原有的 type，避免同名实体被错误改了类型）。Cypher 使用 `$param` 参数化查询，防止注入。

**面试考点**: "为什么用 MERGE 不用 CREATE？"——MERGE 保证幂等性，多次入库同一文档不会重复建节点。CREATE 会导致图谱中有多个"张三"节点，关系断裂。

## 7. GraphRAG 管道 (services/graph_rag.py)

**文件**: `code/python/services/graph_rag.py`

GraphRAG 混合检索的完整 6 步管线实现，是本项目技术含量最高的模块。

```python
class GraphRAGPipeline:
    async def retrieve(self, query: str, top_k: int = 10) -> list[GraphRAGContext]:
        # Step 1-2: 并行检索
        vector_results = await self._vector_search(query, top_k=top_k)
        entities = await self._entity_linking(query)               # LLM提取实体名

        # Step 3-4: 图谱检索
        subgraph_results = await self._subgraph_search(entities)    # 2跳子图遍历
        path_results = await self._path_search(entities)            # 实体间最短路径

        all_results = vector_results + subgraph_results + path_results

        # Step 5: 社区摘要
        if subgraph_results:
            community_ctx = await self._community_summary(subgraph_results)
            all_results.append(community_ctx)

        # Step 6: 交叉重排序 + 去重
        return self._cross_rerank(all_results, query)[:top_k]
```

**交叉重排序的权重设计**（`services/graph_rag.py:220`）：

```python
@staticmethod
def _cross_rerank(contexts, query):
    weight_map = {
        "vector": 1.0,       # 基线：语义匹配的通用能力
        "subgraph": 1.15,    # 结构化关系比语义相似更精准
        "path": 1.25,        # 推理路径最有价值，直接给出推理链
        "community": 1.1,    # 高层概览，补充细节不足
    }
    for ctx in contexts:
        ctx.score *= weight_map.get(ctx.source_type, 1.0)

    # 前80字符去重（相同内容的片段只保留得分最高的）
    seen: set[str] = set()
    unique = []
    for ctx in contexts:
        key = ctx.content[:80]
        if key not in seen:
            seen.add(key)
            unique.append(ctx)

    unique.sort(key=lambda c: c.score, reverse=True)
    return unique
```

**设计要点**: 路径推理给最高权重（1.25）因为它直接回答多跳问题——"张三负责的产品"需要张三→产品的关系链，向量检索只能在语义上"猜"。子图遍历（1.15）提供结构化上下文。向量检索（1.0）是覆盖面最广的基线不能降权。

**面试考点**: "权重 1.25 是怎么调出来的？"——经验值。逻辑是路径推理链 > 子图关系 > 语义相似 > 摘要概览。实际使用中需要根据数据集的检索效果微调。1.25 不是最优值，但它体现了设计者对不同信息来源价值的判断。

## 8. 问答 Agent (agents/qa_agent.py)

**文件**: `code/python/agents/qa_agent.py`

问答是用户感知最直接的模块。完整的 6 步管线在 `answer()` 方法中呈现。

```python
class QAAgent:
    async def answer(self, question: str) -> QAResult:
        intent = await self._classify_intent(question)                # ① 意图分类
        rewritten = await self._rewrite_query(question)               # ② 查询改写

        vector_contexts = await self._vector_retrieve(rewritten)      # ③ 向量检索
        graph_contexts = await self._graph_retrieve(question, rewritten)  # ④ 图谱检索

        all_contexts = self._hybrid_rerank(vector_contexts + graph_contexts)  # ⑤ 混合重排
        top_contexts = all_contexts[:8]                               # 取 top-8

        answer_text, reasoning = await self._generate_answer(         # ⑥ 生成答案
            question, top_contexts, intent
        )
        return QAResult(
            question=question, answer=answer_text,
            contexts=top_contexts, intent=intent,
            confidence=self._calc_confidence(top_contexts),
            reasoning_steps=reasoning,
        )
```

**混合重排序**（图谱结果权重更高）：

```python
@staticmethod
def _hybrid_rerank(contexts):
    weight_map = {"vector": 1.0, "graph": 1.2, "hybrid": 1.1}
    for ctx in contexts:
        ctx.score *= weight_map.get(ctx.retrieval_type, 1.0)
    # 前100字符去重 → 降序排列
    unique.sort(key=lambda c: c.score, reverse=True)
    return unique
```

**设计要点**: 向量检索和图谱检索虽然代码上是串行调用，但它们在逻辑上是独立的（没有数据依赖），可以改为 `asyncio.gather` 并行执行。

**面试考点**: "为什么 graph 给 1.2 倍权重？"——图谱结果来自 Neo4j 结构化查询（如 "张三 WORKS_AT 腾讯"），是确定性的结构化事实。向量检索是语义近似的概率性结果。结构化事实比语义近似更可靠。

## 9. 编排引擎 (orchestrator/graph.py)

**文件**: `code/python/orchestrator/graph.py`

LangGraph StateGraph 编排三条流水线——这是连接 API 和 Agent 的中枢。

```python
def _build_ingest_graph(doc_parser, extractor, vector_store, knowledge_graph):
    async def parse_documents(state):    # 节点1: 文档解析
        chunks = await doc_parser.parse_batch(state.get("file_paths", []))
        return {**state, "chunks": chunks}

    async def extract_knowledge(state):  # 节点2: 知识抽取
        extractions = await extractor.extract(state.get("chunks", []))
        return {**state, "extractions": extractions}

    async def store_vectors(state):      # 节点3: 向量入库
        count = await vector_store.add_chunks(state.get("chunks", []))
        return {**state, "vectors_stored": count}

    async def store_graph(state):        # 节点4: 图谱入库
        for ext in state.get("extractions", []):
            for ent in ext.entities:
                await knowledge_graph.upsert_entity(ent)
            for rel in ext.relations:
                await knowledge_graph.add_relation(rel)
        return {**state, "entities_stored": entity_count}

    # 构建有向图: parse → extract → store_vectors → store_graph → END
    graph = StateGraph(dict)              # StateGraph 的 state 类型是普通 dict
    graph.add_node("parse", parse_documents)
    graph.add_node("extract", extract_knowledge)
    graph.add_node("store_vectors", store_vectors)
    graph.add_node("store_graph", store_graph)
    graph.set_entry_point("parse")        # 入口节点
    graph.add_edge("parse", "extract")    # 有向边
    graph.add_edge("extract", "store_vectors")
    graph.add_edge("store_vectors", "store_graph")
    graph.add_edge("store_graph", END)
    return graph.compile()                # 编译为可执行的 Runnable
```

**更新流是唯一使用条件路由的流水线**：

```python
def _build_update_graph(update_agent):
    def should_continue(state):           # 条件判断函数
        failed = [r for r in state["results"] if not r.success]
        return "retry" if failed else "done"

    graph = StateGraph(dict)
    graph.add_node("process", process_updates)
    graph.add_node("retry", retry_failed)
    graph.set_entry_point("process")
    graph.add_conditional_edges(           # ← 唯一使用条件边的地方
        "process", should_continue,
        {"retry": "retry", "done": END}
    )
    graph.add_edge("retry", END)
    return graph.compile()
```

**设计要点**: `StateGraph(dict)` 使用普通 dict 而非 TypedDict——灵活性更高，节点可以自由添加字段。代价是没有编译期类型检查。

**面试考点**: "为什么不把三条流水线合成一个大的 StateGraph？"——三条流水线的 state 结构不同（入库有 chunks、问答有 question、更新有 changes），合在一起状态管理会很混乱。分开三个 graph 是更好的职责分离。

```python
    result = await workflows["ingest"].ainvoke({"file_paths": [save_path]})
    chunks = result.get("chunks", [])
    extractions = result.get("extractions", [])

    # ③ 统计结果并返回
    return IngestResponse(
        file_name=file.filename,
        chunks_count=len(chunks),
        entities_count=sum(len(e.entities) for e in extractions),
        relations_count=sum(len(e.relations) for e in extractions),
        status="success",
    )
```

**核心端点 — 智能问答**：

```python
@app.post("/api/qa/ask", response_model=QuestionResponse, tags=["智能问答"])
async def ask_question(req: QuestionRequest):
    result = await workflows["qa"].ainvoke({"question": req.question})
    qa_result = result.get("result")
    return QuestionResponse(
        question=qa_result.question,
        answer=qa_result.answer,
        confidence=qa_result.confidence,
        intent=qa_result.intent.value,
        sources=[{"content": c.content[:200], ...} for c in qa_result.contexts],
        reasoning_steps=qa_result.reasoning_steps,
    )
```

**设计要点**: FastAPI 的 `response_model` 自动生成 Swagger 文档和响应校验。`lifespan` 异步上下文管理器在启动时初始化所有依赖，关闭时释放资源。

**面试考点**: "为什么要用 lifespan 而不是 `@app.on_event("startup")`？"——`on_event` 在 FastAPI 新版本中已废弃。lifespan 是推荐的异步生命周期管理方式，支持 yield 两侧的 setup/teardown 逻辑。
