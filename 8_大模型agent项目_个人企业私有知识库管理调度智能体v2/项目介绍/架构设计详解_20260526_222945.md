# com_agent_chat — 架构设计详解

## 一、整体架构总览

```
                           ┌──────────────────────────────┐
                           │       用户接口层                │
                           │   Web UI (SPA 三Tab)           │
                           │   REST API (FastAPI 7端点)      │
                           │   Swagger 自动文档              │
                           └──────────────┬───────────────┘
                                          │ HTTP
                           ┌──────────────▼───────────────┐
                           │     编排引擎层 (LangGraph)      │
                           │                               │
                           │  ┌─────────────────────────┐  │
                           │  │    Ingest Pipeline       │  │
                           │  │  parse → extract → store │  │
                           │  └─────────────────────────┘  │
                           │  ┌─────────────────────────┐  │
                           │  │    QA Pipeline            │  │
                           │  │  answer → END             │  │
                           │  └─────────────────────────┘  │
                           │  ┌─────────────────────────┐  │
                           │  │    Update Pipeline        │  │
                           │  │  process → retry → END    │  │
                           │  └─────────────────────────┘  │
                           └──────────────┬───────────────┘
                                          │
              ┌───────────────────────────┼────────────────────────────┐
              │                           │                            │
     ┌────────▼────────┐    ┌─────────────▼──────────┐    ┌───────────▼────────┐
     │   Agent 层       │    │      服务层              │    │   基础设施层         │
     │                  │    │                          │    │                    │
     │ DocParserAgent   │───▶│ VectorStoreService       │───▶│ ChromaDB / PGVector│
     │ KnowledgeExtract │───▶│ KnowledgeGraphService    │───▶│ Neo4j              │
     │ QAAgent          │───▶│ GraphRAGPipeline         │───▶│ Kafka              │
     │ KnowledgeUpdate  │───▶│ CDCProcessor              │    │ EmbeddingWorker    │
     │                  │    │ MultimodalService         │    │ (子进程)            │
     └──────────────────┘    └──────────────────────────┘    └────────────────────┘
```

**分层说明**：
- **接口层**：对外的 HTTP API 和 Web UI，不包含业务逻辑，只做请求校验和响应格式化
- **编排层**：定义多 Agent 协作的流程（LangGraph StateGraph），管理状态传递和条件路由
- **Agent 层**：业务逻辑的核心，每个 Agent 独立完成一个子领域的任务
- **服务层**：数据访问抽象，封装向量库/图谱/CDC/多模态的具体实现
- **基础设施层**：外部依赖（数据库、消息队列、Embedding 模型）

分层间的依赖方向是单向的：上层依赖下层，下层不感知上层。

## 二、为什么选 LangGraph StateGraph 做编排？

### 编排模式对比

| 维度 | Pipeline | DAG (有向无环图) | 状态机 | LangGraph StateGraph |
|------|----------|-----------------|--------|---------------------|
| 控制粒度 | 低（纯线性） | 中（支持分支汇聚） | 中（有限状态数） | 高（显式图+条件路由+状态） |
| 并行支持 | 无 | 有（fork/join） | 有限 | 有（条件边） |
| 状态管理 | 手动 | 部分 | 内置 | 内置（checkpointer） |
| 生产就绪 | 低 | 中 | 中 | 高（LangSmith 集成） |
| 学习曲线 | 1 天 | 2-3 天 | 2-3 天 | 2-5 天 |

### 本项目选择 LangGraph 的理由

1. **精确控制流程**：入库流的 parse→extract→store 需要严格串行（下游依赖上游数据），更新流需要条件路由（失败→retry→成功→END）。StateGraph 的显式边定义让这些控制流一目了然。

2. **状态传递内置**：每个节点接收 state dict、返回更新后的 state dict，不需要手动管理全局变量或数据库中间表。

3. **生产可观测性**：LangSmith 集成让每步的输入输出可追踪，方便调试。

### 为什么不用 CrewAI？

CrewAI 基于角色定义（Role-Play），让 Agent 通过对话协作，适合开放式任务。但本项目的流程有明确的输入输出依赖（解析的输出必须是抽取的输入），需要精确的数据传递，对话式协作引入不必要的交流开销。而且 CrewAI 的状态管理是基础的，不支持 checkpointer。

### 为什么不用 AutoGen？

AutoGen 的对话模式更适合多 Agent 讨论类任务（如代码审查、头脑风暴），不适合本项目的流水线式任务。AutoGen 的控制粒度偏研究导向，生产化程度不如 LangGraph。

## 三、4 个模块的设计

### 拆分原则

4 个 Agent 的拆分遵循"按知识生命周期划分"原则：

| 生命周期阶段 | Agent | 输入 | 输出 |
|-------------|-------|------|------|
| 知识获取 | DocParserAgent | 文件路径 | DocumentChunk 列表 |
| 知识提炼 | KnowledgeExtractAgent | DocumentChunk 列表 | 实体/关系/事件 |
| 知识消费 | QAAgent | 自然语言问题 | 答案+来源+置信度 |
| 知识维护 | KnowledgeUpdateAgent | 变更事件 | 增量更新结果 |

### 基类设计

项目没有抽象基类——4 个 Agent 是独立的类，各自有自己的公共API。这种设计比抽象基类更灵活：
- 新增 Agent 不需要遵循固定接口
- 每个 Agent 的 public API 只暴露业务需要的方法（如 `parse()`、`extract()`、`answer()`、`process_change()`）
- 通过构造函数注入依赖而非通过基类约束

```python
# QAAgent 的依赖注入设计
class QAAgent:
    def __init__(self, vector_store=None, knowledge_graph=None):
        self.vector_store = vector_store
        self.knowledge_graph = knowledge_graph
        self.llm = ChatOpenAI(...)  # 自己管理 LLM 客户端
```

### 模块间通信方式

| 通信场景 | 方式 | 为什么 |
|---------|------|--------|
| 入库流中 Agent 间传递数据 | LangGraph state dict | 编排器负责数据流转，Agent 不直接通信 |
| QAAgent 查询向量库 | 直接方法调用 `vector_store.search()` | 查询类操作适合同步返回 |
| QAAgent 查询图谱 | 直接方法调用 `knowledge_graph.execute_cypher()` | 同上 |
| CDC 事件流入 | Kafka 消息或 Watchdog 事件 | 异步事件适合消息队列或文件监听 |
| Embedding 请求 | multiprocessing.Queue | 跨进程通信，避免 C 扩展 segfault |

## 四、并行执行的设计（为何 store_vectors 和 store_graph 不并行）

在入库流水线中，`store_vectors` 和 `store_graph` 之间当前是串行关系（`store_vectors → store_graph → END`）。从逻辑上讲，向量存储和图谱存储没有数据依赖，完全可以并行执行。

**当前的设计**：

```python
graph.add_edge("extract", "store_vectors")  # 串行
graph.add_edge("store_vectors", "store_graph")  # 串行
graph.add_edge("store_graph", END)
```

**为什么没有用并行**：

LangGraph 的并行机制是通过从同一个节点引出两条边来实现的。如果改为并行：

```python
graph.add_edge("extract", "store_vectors")  # 分支1
graph.add_edge("extract", "store_graph")     # 分支2
# 需要一个汇聚节点合并两个分支的结果
```

并行需要额外的汇聚节点（merge state），复杂度增加。当前项目处于演示阶段，入库延迟不是瓶颈（核心延迟在 LLM 调用和 Embedding 推理），串行存储的额外时间可以忽略。如果文档量增加到需要并行处理时，改为并行是直接的——只需要加一个汇聚节点。

**面试讲法**：
> "当前入库流中向量和图谱是串行存储的。代码架构上改为并行只需要改两条边——因为它们之间没有数据依赖。我没有做并行是因为当前阶段入库延迟的瓶颈在 LLM 调用（知识抽取），而不是存储。过早优化是万恶之源。"

## 五、更新流水线的条件路由设计

### 流程图

```
                    ┌──────────┐
                    │ process   │  ← 入口：批量处理变更
                    └─────┬─────┘
                          │
                          ▼
              ┌───────────────────────┐
              │ should_continue(state) │  ← 条件判断
              └───────────┬───────────┘
                          │
               ┌──────────┴──────────┐
               │                     │
         有失败的更新            全部成功
               │                     │
               ▼                     ▼
          ┌─────────┐           ┌─────────┐
          │  retry  │           │   END   │
          └────┬────┘           └─────────┘
               │
               ▼
              只重试失败的变更 → END
```

### 终止条件

1. **成功终止**：`should_continue` 发现所有 `UpdateResult.success=True` → 走向 END
2. **失败后终止**：`retry` 节点执行后直接走向 END（`graph.add_edge("retry", END)`），不管重试是否成功
3. **设计意图**：只重试一次，避免无限循环。生产环境可以改为重试 N 次或指数退避。

### 防止异常的关键代码

```python
def should_continue(state: dict) -> str:
    results = state.get("results", [])
    failed = [r for r in results if not r.success]
    if failed:
        return "retry"    # 有失败→重试
    return "done"          # 全成功→结束

# 编排器定义
graph.add_conditional_edges(
    "process", should_continue,
    {"retry": "retry", "done": END}
)
graph.add_edge("retry", END)  # 重试后不管结果，直接结束
```

**面试讲法**：
> "更新流是唯一使用条件路由的流水线。核心逻辑在 `should_continue` 函数——检查是否有失败的更新任务。有失败就走 retry 节点重试，全部成功就走 END。重试只做一次，避免无限循环。如果需要更鲁棒的重试策略，可以引入指数退避和最大重试次数。"

## 七、扩展性设计

### 如何新增一个 Agent

1. 在 `agents/` 下创建新的 Agent 类，如 `agents/summarize_agent.py`
2. 实现公共 API 方法
3. 在 `agents/__init__.py` 中导出
4. （如需新流水线）在 `orchestrator/graph.py` 中添加新流水线构建函数
5. （如需新 API 端点）在 `api/main.py` 中添加新路由

```python
# 示例：添加文档摘要 Agent
# agents/summarize_agent.py
class SummarizeAgent:
    def __init__(self, vector_store=None):
        self.llm = ChatOpenAI(...)
        self.vector_store = vector_store

    async def summarize(self, doc_id: str) -> str:
        chunks = await self.vector_store.get_by_doc_id(doc_id)
        prompt = f"请总结以下文档内容：\n{chunks}"
        resp = await self.llm.ainvoke(prompt)
        return resp.content
```

### 如何替换向量库实现

1. 在 `services/vector_store.py` 的 `VectorStoreService` 中添加新后端的 `_init_xxx()` 方法
2. 在 `init()` 方法中添加条件分支
3. 在 `search()`、`add_chunks()`、`delete_by_doc_id()` 中添加对应实现
4. 在 `config/settings.py` 中添加新后端的配置项

### 总结

- **改几个文件**：新增 Agent 只需改 3 个文件（Agent 文件 + `__init__.py` + `graph.py`），新增 API 端点只需改 `api/main.py`
- **不影响什么**：Agent 之间完全解耦，新增 Agent 不影响现有 Agent 的逻辑
- **依赖注入**：服务通过构造函数注入，替换实现只需在 `api/main.py` 的 lifespan 中换一个实例
