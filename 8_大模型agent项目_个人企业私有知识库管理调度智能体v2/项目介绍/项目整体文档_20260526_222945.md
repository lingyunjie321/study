# com_agent_chat — 完整项目文档

## 项目简介

**com_agent_chat** 是一个基于4-Agent混合编排的企业知识调度管理系统，实现从多模态文档入库、知识图谱自动构建、GraphRAG混合检索问答到CDC增量更新的知识全生命周期管理。

**一句话描述**：上传一份PDF，用自然语言提问，AI综合向量语义搜索和知识图谱多跳推理给你精准答案，文档更新后知识库自动同步。

### 核心功能

| 功能 | 说明 | 解决什么问题 |
|------|------|-------------|
| 多模态文档解析 | 自动识别并解析PDF/图片/Excel/CSV/Markdown等格式 | 企业文档格式多样，传统系统只能处理纯文本 |
| 知识图谱自动构建 | 从文档中抽取实体、关系、事件，自动构建Neo4j图 | 纯向量检索无法理解"张三的老板的公司"这种多跳关系 |
| GraphRAG混合问答 | 向量语义检索+图谱多跳推理+交叉重排序，带来源引用 | 单独用向量或图谱检索各有盲区，融合后互补 |
| CDC增量更新 | 文件变更自动检测，只更新变化部分，版本可回溯 | 1000个文档改5个，传统全量重建要30分钟，CDC只需30秒 |
| Web可视化界面 | 问答聊天、文档上传、系统仪表盘三合一SPA | 降低使用门槛，非技术人员也能操作 |

- **项目类型**：企业级AI知识管理平台
- **开发语言**：Python 3.12+
- **状态**：功能完整，可本地运行演示

### 核心数据模型

```python
# 文档块 — 向量检索的基本单位
@dataclass
class DocumentChunk:
    content: str           # 文本内容（512字符窗口）
    doc_id: str            # 源文档SHA256前16位
    chunk_index: int       # 在文档中的序号
    doc_type: DocType      # PDF/IMAGE/TABLE/TEXT/MARKDOWN
    metadata: dict         # 来源、字符位置等
    embedding: list[float] | None  # 向量表示（384维）

# 知识三元组 — 图谱检索的基本单位
@dataclass
class Entity:
    name: str              # 实体名称，如"张三"
    type: str              # Person/Organization/Product等8种
    description: str       # 简短描述
    properties: dict       # 扩展属性

@dataclass
class Relation:
    head: str              # 头实体
    relation: str          # 关系类型（8种）
    tail: str              # 尾实体
    confidence: float      # 置信度 0-1

# 问答结果
@dataclass
class QAResult:
    question: str          # 原始问题
    answer: str            # 生成的答案
    contexts: list         # 检索到的上下文（带来源）
    intent: QueryIntent    # 意图分类
    confidence: float      # 置信度分数
    reasoning_steps: list  # 推理步骤日志
```

## 核心组件详解

### 1. DocParserAgent — 文档解析Agent

**文件**: `agents/doc_parser_agent.py` (287行)

**职责**：将多模态文档转换为统一的文本块，供后续知识抽取和向量化使用。

```python
class DocParserAgent:
    SUPPORTED_EXTENSIONS = {
        ".pdf": DocType.PDF, ".png": DocType.IMAGE,
        ".jpg": DocType.IMAGE, ".csv": DocType.TABLE,
        ".xlsx": DocType.TABLE, ".txt": DocType.TEXT,
        ".md": DocType.MARKDOWN,
    }
    CHUNK_SIZE = 512     # 每个chunk 512字符
    CHUNK_OVERLAP = 64   # 相邻chunk重叠64字符
```

**内部处理流程**：

```
文件路径
  │
  ▼
_classify(file_path)      ← 按扩展名识别类型
  │
  ├─ PDF → _parse_pdf()  → PyPDF2提取文字 → 失败则LLM视觉兜底
  ├─ IMG → _parse_image()→ Tesseract OCR + LLM视觉描述
  ├─ CSV  → _parse_csv() → DictReader结构化提取
  ├─ XLSX → _parse_excel()→ openpyxl逐表提取
  └─ TXT/MD → _parse_text() → 直接读取
  │
  ▼
_chunk_texts()            ← 512字符窗口 + 64字符重叠滑动
  │
  ▼
list[DocumentChunk]       ← 带doc_id、chunk_index、metadata
```

**关键特性**：
- PDF解析失败时自动降级为LLM视觉理解（`_pdf_vision_fallback`）
- 图片同时使用OCR和LLM视觉描述，互补信息
- 表格结构化后转为"列名: 值"的文本表示，保留语义
- 每个chunk携带来源文件、字符位置等完整元数据

### 2. KnowledgeExtractAgent — 知识抽取Agent

**文件**: `agents/knowledge_extract_agent.py` (211行)

**职责**：从文档块中自动识别实体、关系和事件，将非结构化文本转为结构化知识三元组。

**内部处理流程**：

```
list[DocumentChunk]
  │
  ▼
批量处理 (BATCH_SIZE=5)
  │
  ├─ 每个chunk → LLM + JSON Prompt
  │   提取: entities[], relations[], events[]
  │
  ▼
_deduplicate()            ← 跨chunk去重
  │  实体: 按 (name, type) 去重
  │  关系: 按 (head, relation, tail) 去重
  │
  ▼
list[ExtractionResult]    ← 去重后的结构化知识
```

**关键特性**：
- 8种实体类型：Person, Organization, Location, Product, Technology, Concept, Event, Time
- 8种关系类型：belongs_to, works_at, located_in, developed_by, related_to, part_of, uses, depends_on
- 跨chunk去重避免同一实体在多处出现时重复建节点
- 每次抽取返回confidence置信度，用于后续重排序的权重参考

### 3. QAAgent — 问答Agent

**文件**: `agents/qa_agent.py` (287行)

**职责**：接收自然语言问题，完成意图识别→查询改写→并行检索→混合排序→答案生成的全链路。

**内部处理流程**：

```
用户问题
  │
  ▼
_classify_intent()         ← LLM分类: factoid/analytical/comparative/procedural/exploratory
  │
  ▼
_rewrite_query()           ← LLM改写: 提取实体+关键词+生成1-3个检索query
  │
  ├────────────────────并行───────────────────┐
  ▼                                           ▼
_vector_retrieve()                          _graph_retrieve()
  每个query查向量库top-5                        LLM生成Cypher→Neo4j执行
  (语义相似度)                                  (关系路径)
  │                                           │
  └────────────────────合并───────────────────┘
  │
  ▼
_hybrid_rerank()           ← 权重: graph×1.2, vector×1.0
  │                           前100字符去重
  ▼
取top-8
  │
  ▼
_generate_answer()         ← LLM基于上下文生成答案+来源引用
  │
  ▼
QAResult {answer, confidence, sources, reasoning_steps}
```

**关键特性**：
- 向量检索和图谱检索并行执行，减少等待时间
- 图谱结果权重1.2倍（结构化关系比语义相似更精准）
- 答案必带来源引用`[来源: xxx]`
- 知识库为空时不强行编造，直接告知用户

### 4. KnowledgeUpdateAgent — 知识更新Agent

**文件**: `agents/knowledge_update_agent.py` (271行)

**职责**：监听文档变更事件，增量更新向量库和知识图谱，支持版本回溯。

**内部处理流程**：

```
变更事件 (文件系统Watchdog / Kafka消息)
  │
  ▼
detect_changes()           ← SHA256哈希对比，三态判定：CREATED/MODIFIED/DELETED
  │
  ├─ CREATED → _handle_create()  → 解析→向量化→抽取→写入图谱
  ├─ MODIFIED→ _handle_modify()  → 先删旧数据→再走CREATE流程
  └─ DELETED → _handle_delete()  → 从向量库和图谱中删除
  │
  ▼
_bump_version()            ← 每次更新实体版本号+1
  │
  ▼
UpdateResult {vectors_added, entities_updated, processing_time_ms, success}
```

**关键特性**：
- 双模式：Watchdog实时监听文件系统 + Kafka消费CDC事件
- 差量分析：对比新旧内容，>30%变化视为大改（全量重建），否则小改（增量更新）
- 版本管理：每个知识节点带version+timestamp，支持回滚
- 失败重试：LangGraph更新流水线内置条件路由，失败自动retry

### 5. GraphRAGPipeline — 混合检索管道

**文件**: `services/graph_rag.py` (234行)

**职责**：GraphRAG的完整实现，6步流水线融合向量和知识图谱检索。

```
Query
  │
  ├── Step 1: _vector_search() → 向量语义检索 top-k
  ├── Step 2: _entity_linking() → LLM提取query中实体
  ├── Step 3: _subgraph_search()→ 从实体出发2跳Neo4j遍历
  ├── Step 4: _path_search()    → 实体对间shortestPath推理
  ├── Step 5: _community_summary() → LLM子图摘要
  │
  └── Step 6: _cross_rerank()
       权重: vector×1.0, subgraph×1.15, path×1.25, community×1.1
       前80字符去重 → 按分排序 → 返回top_k
```

## 数据库架构

### ChromaDB（向量库）

```
Collection: knowledge_chunks
  ├── ids: 自动生成UUID
  ├── embeddings: 384维向量 (HNSW索引 + cosine距离)
  ├── documents: 文本内容
  └── metadatas: {source, doc_id, chunk_index, doc_type, char_start, char_end}

向量检索参数:
  - 距离度量: cosine
  - 索引算法: HNSW
  - 默认top_k: 5
```

### Neo4j（知识图谱）

```cypher
-- 节点
(:Entity {
    name: "张三",           -- 实体名（有索引）
    type: "Person",         -- 实体类型（有索引）
    description: "腾讯CEO", -- 描述
    version: 3,             -- 版本号（每次更新+1）
    source: "/docs/年报.pdf",-- 来源文件（有索引）
    created_at: 1715000000, -- Unix时间戳
    updated_at: 1715086400
})

-- 关系（8种类型）
(:Entity)-[:WORKS_AT {confidence: 0.95, source: "...", updated_at: ...}]->(:Entity)
(:Entity)-[:DEVELOPED_BY]->(:Entity)
(:Entity)-[:BELONGS_TO]->(:Entity)
(:Entity)-[:LOCATED_IN]->(:Entity)
(:Entity)-[:RELATED_TO]->(:Entity)
(:Entity)-[:PART_OF]->(:Entity)
(:Entity)-[:USES]->(:Entity)
(:Entity)-[:DEPENDS_ON]->(:Entity)

-- 索引
CREATE INDEX FOR (n:Entity) ON (n.name)
CREATE INDEX FOR (n:Entity) ON (n.type)
CREATE INDEX FOR (n:Entity) ON (n.source)
```

### 连接管理

- Neo4j使用`AsyncGraphDatabase`驱动，应用启动时建立连接池，关闭时释放
- ChromaDB使用`PersistentClient`，数据持久化到`chroma_data/`目录
- Embedding模型在独立子进程中加载，通过`multiprocessing.Queue`通信，避免PyTorch+Pytorch的C扩展在async环境下segfault

## 常见问题与调试

### Q1: Neo4j连接失败

**症状**：启动时报错 `Unable to retrieve routing information`

**原因**：Neo4j服务未启动或密码不正确

**解决方案**：
```bash
# 确认Neo4j运行中
docker-compose ps neo4j
# 查看日志
docker-compose logs neo4j
# 首次启动需要等待约20秒初始化
```

### Q2: ChromaDB在async环境崩溃

**症状**：`Segmentation fault` 或 `Access violation`

**原因**：ChromaDB的C扩展与Python asyncio事件循环冲突

**解决方案**：项目已内置处理——使用线程池隔离ChromaDB操作，并手动维护向量计数。如需完全避免，设置环境变量：
```bash
export DISABLE_LOCAL_EMBEDDINGS=1  # 只用LLM，不加载本地Embedding模型
```

### Q3: Embedding模型加载超时

**症状**：启动时卡在 `[embedding_worker] Model loaded`

**原因**：首次下载`text2vec-base-chinese`模型需要时间，或网络不通

**解决方案**：
```bash
# 手动预下载模型
python -c "from sentence_transformers import SentenceTransformer; SentenceTransformer('shibing624/text2vec-base-chinese')"
# 或设置镜像源
export HF_ENDPOINT=https://hf-mirror.com
```

### Q4: Kafka连接失败不影响核心功能

**症状**：日志中出现Kafka连接错误

**原因**：Kafka用于CDC增量更新，不影响文档入库和问答

**解决方案**：核心功能不需要Kafka，可暂时忽略。如需增量更新，确保`docker-compose up -d kafka`已执行。

### Q5: PDF中文解析乱码

**症状**：PDF中文变成乱码或空白

**原因**：PDF使用了非标准字体编码

**解决方案**：项目已内置LLM视觉兜底——PDF文字提取失败时自动调用DeepSeek多模态能力描述页面内容。

## 性能优化建议

| 瓶颈 | 当前方案 | 优化方向 | 优先级 |
|------|---------|----------|--------|
| Embedding推理 | 本地CPU跑text2vec-base-chinese | 迁移到GPU或使用API Embedding服务 | 🟡中 |
| ChromaDB扩展性 | 嵌入式单机 | 迁移到Milvus（支持百亿级向量+分布式） | 🟢低 |
| LLM调用延迟 | 每次请求同步调用 | 加入语义缓存，相同/相似问题直接返回缓存 | 🔴高 |
| 图谱检索范围 | 2跳遍历LIMIT 50 | 引入图嵌入(GraphSAGE)预计算节点向量 | 🟢低 |
| 文档解析并行度 | 单个文件串行解析 | 多文件并行解析，Celery异步任务队列 | 🟡中 |
| 知识抽取速度 | 每chunk一次LLM调用 | 批量Prompt一次抽取多个chunk | 🟡中 |

## 安全注意事项

- **API密钥保护**：所有LLM API密钥通过环境变量注入，不在代码中硬编码；`.env`文件已加入`.gitignore`
- **Neo4j认证**：默认密码`password`仅用于本地开发，生产环境务必修改
- **文件上传安全**：仅允许在`UPLOAD_DIR`目录内操作，需添加文件类型白名单校验（当前未限制）
- **注入防护**：Cypher查询使用参数化方式（`$param`），避免Cypher注入；但LLM生成的Cypher未做安全审查，生产环境需增加语法校验
- **敏感信息**：文档中出现的API密钥、密码等统一替换为`<敏感信息>`占位符
