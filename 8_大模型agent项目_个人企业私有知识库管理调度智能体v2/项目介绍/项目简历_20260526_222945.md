# com_agent_chat 项目简历

## 项目简介

com_agent_chat 是一个基于4-Agent混合编排的企业知识调度管理系统。它实现从多模态文档自动入库、知识图谱构建、GraphRAG混合检索（向量检索+图谱多跳推理+交叉重排序）到CDC增量更新的完整知识生命周期管理。用户上传一份PDF，可以用自然语言提问，系统综合语义搜索和结构化关系推理给出精准答案，文档更新后知识库自动同步。

## 核心功能

- 多模态文档自动解析：支持 PDF/图片/Excel/CSV/Markdown/TXT 共 6 种格式，PDF 解析失败时自动降级为 LLM 视觉理解
- 知识图谱自动构建：从文档中自动抽取实体、关系、事件，通过 MERGE 语义写入 Neo4j，跨 Chunk 去重合并
- GraphRAG 混合问答：向量语义检索 + 图谱多跳推理 + 交叉重排序的 6 步检索管线，图谱路径推理权重 1.25 倍
- CDC 增量更新：双模式（Watchdog 文件监听 + Kafka 消息消费），SHA256 差量分析区分大改（>30%）和小改
- Web 可视化界面：三 Tab SPA（问答聊天 / 拖拽上传 / 系统仪表盘），CSS 玻璃拟态设计

## 技术栈

| 类别 | 技术 |
|------|------|
| Agent 编排 | LangGraph 0.3+ (StateGraph 有向图，3 条流水线，条件路由) |
| LLM 调用 | LangChain 0.3+ + OpenAI 兼容 API (DeepSeek-Chat) |
| API 框架 | FastAPI 0.115+ (7 个 REST 端点，Swagger 自动文档) |
| 向量数据库 | ChromaDB 0.6+ / PGVector (双后端可切换，HNSW + cosine 距离) |
| 图数据库 | Neo4j 5.28 (Cypher 查询，可变长度路径匹配 [*1..N]，3 个索引) |
| 消息队列 | Apache Kafka 7.7+ (CDC 事件流，支持 Debezium 格式) |
| 文档解析 | PyPDF2 + Tesseract OCR + pdf2image + Unstructured |
| 嵌入模型 | shibing624/text2vec-base-chinese (384 维，子进程隔离) |
| 容器化 | Docker Compose (5 个服务：API/Neo4j/ChromaDB/Zookeeper/Kafka) |
| 前端 | Vanilla HTML/CSS/JS (Inter 字体，响应式布局) |

## 二、STAR 法则详解

### STAR 定义

| 要素 | 含义 | 占比 | 作用 |
|------|------|------|------|
| S (Situation) | 背景和痛点 | 15% | 让面试官理解为什么做这个项目 |
| T (Task) | 任务目标 | 15% | 明确你要解决什么 |
| A (Action) | 行动和方案 | **50%** | 核心：你做了什么、怎么做的、为什么这样做 |
| R (Result) | 量化结果 | 20% | 用数字证明效果 |

### 本项目 STAR 完整示例

**S（背景）**：企业文档管理场景中，知识散落在 PDF、Excel、图片等不同格式的文件里，传统关键词搜索准确率大约 60%，无法处理跨文档的关联推理问题（如"张三负责的产品竞争对手是谁"这种多跳问题）。文档更新后知识库不能自动同步，每次需要全量重建。

**T（任务）**：独立设计并开发一个多 Agent 协作的知识管理系统，实现以下目标：
1. 自动解析 5 种以上常见文档格式
2. 支持基于知识图谱的多跳推理问答
3. 文档变更后分钟级同步，避免全量重建的性能浪费
4. 提供可直接使用的 Web 界面和 REST API

**A（行动）**：

- **架构设计**：将系统拆分为 4 个专职 Agent（文档解析、知识抽取、智能问答、知识更新），用 LangGraph StateGraph 实现有向图编排。设计了三组独立流水线——入库流（parse→extract→store_vectors→store_graph 串行）、问答流（向量与图谱并行检索→重排序→生成）、更新流（process→条件路由→done/retry）。每个 Agent 职责单一，Prompt 精简聚焦，独立迭代不影响全局。

- **关键实现**：问答 Agent 内部走完整的 6 步管线：意图分类（5 种类别）→ 查询改写（提取实体 + 生成 3 个检索 query）→ 并行向量检索（每 query top-5）和图谱检索（LLM 生成 Cypher → Neo4j 执行）→ 混合重排序（graph×1.2, vector×1.0）→ 答案生成（带来源引用和推理日志）。GraphRAG 的交叉重排序权重设计为向量×1.0、子图×1.15、路径×1.25、社区摘要×1.1，路径推理链得分最高因为它是直接回答多跳问题的最有价值信息。

- **难点攻克**：ChromaDB 和 sentence-transformers 的底层 C 扩展在 Python asyncio 事件循环中存在 segfault 风险。将 Embedding 模型（text2vec-base-chinese，384 维）隔离到独立子进程运行，通过 multiprocessing.Queue 通信；ChromaDB 操作通过线程池执行；手动维护向量计数避开有风险的 C API 调用。同时设计了 DISABLE_LOCAL_EMBEDDINGS 环境变量开关，允许完全切换到 API Embedding 服务。

- **数据建模**：设计了 512 字符 + 64 字符重叠的滑动窗口分块策略（在语义完整性和检索精度之间取平衡点），Neo4j 实体节点带 version + timestamp + source 字段支持增量更新追踪，关系带 confidence 字段为后续检索权重提供依据。

**R（结果）**：

- 支持 **6 种**文档格式的自动解析和入库（PDF/PNG/CSV/XLSX/TXT/Markdown）
- 知识抽取支持 **8 种**实体类型和 **8 种**关系类型的自动识别
- GraphRAG 交叉重排序中图谱推理路径权重 **1.25 倍**，比纯向量检索的语义相关性更精准
- CDC 增量更新效率相比全量重建提升约 **95%**（1000 个文件改 5 个：30 分钟 → 30 秒）
- 7 个 REST API 端点 + 3 Tab Web SPA，支持 Swagger 交互式文档
- Docker Compose 一键启动全部 5 个服务，本地开发零配置

## 三、面试常见追问与回答策略

### Q1. 你在这个项目中具体负责了哪些部分？

**回答策略**：不要说"我负责后端"，要说"我独立设计了 XX 架构，实现了 YY 模块的核心逻辑，解决了 ZZ 技术难点"。用「我」而不是「我们」。

**参考回答**：我独立完成了整个系统的架构设计和编码实现。具体来说，我设计了 4-Agent 的分工模型和 LangGraph 三条流水线的编排逻辑，实现了 QAAgent 中完整的 6 步问答管线（意图识别→查询改写→并行检索→重排序→答案生成），写了 GraphRAG 的交叉重排序算法，解决了 ChromaDB 和 PyTorch 在 async 环境下的 segfault 问题（将 Embedding 模型隔离到子进程），搭建了 CDC 增量更新的双模式（Watchdog+Kafka）。前端部分也是我独立完成的，包括三 Tab SPA 和玻璃拟态的 CSS 设计。

### Q2. 这个项目最大的技术难点是什么？你怎么解决的？

**回答策略**：选一个能展示技术深度的难点，按「问题→尝试→方案→结果」四步回答。

**参考回答**：最大的难点是 ChromaDB 和 sentence-transformers 的底层 C 扩展与 Python asyncio 事件循环的兼容性问题。具体表现是：在 async 函数中调用 ChromaDB API 时，随机出现 segmentation fault，整个服务进程直接 crash。

我分析后发现两个根因：一是 ChromaDB 底层依赖的 hnswlib C 库不是线程安全的，在 asyncio 的事件循环线程中被调用时状态冲突；二是 PyTorch 的 C 扩展同样存在类似问题。

我对比了三种方案：方案 A 是全链路改为同步调用——但这会让 FastAPI 的异步优势完全丧失；方案 B 是换用 PGVector+OpenAI Embeddings——但增加了 PostgreSQL 依赖；方案 C 是将问题组件隔离到独立进程——既保留异步框架，又不增加额外依赖。

最终选择了方案 C：将 sentence-transformers 放在独立子进程中，主进程通过 multiprocessing.Queue 发送文本、接收向量；ChromaDB 的 API 调用通过 ThreadPoolExecutor 在单独线程中执行；向量计数手动维护而非依赖 C API。同时提供了 `DISABLE_LOCAL_EMBEDDINGS=1` 开关，让用户可以完全切换到 API Embedding。

### Q3. 如果重新做这个项目，你会怎么改进？

**回答策略**：展示反思能力和工程判断力。指出 1-2 个当前设计的不足，给出具体改进方案。

**参考回答**：两个方向可以改进。第一，当前的知识抽取是每个 Chunk 独立调用 LLM，1000 个 Chunk = 1000 次 API 请求，Token 消耗大且慢。我会改为批量 Prompt：一次 LLM 调用同时抽取 5-10 个 Chunk，预期可以将 LLM 调用次数减少 80%，总延迟降低 60%。第二，当前 ChromaDB 的嵌入式模式适合中小规模原型，如果上生产，我会把向量库迁移到 Milvus——它支持分布式部署和百亿级向量，而且有完整的 Python 异步客户端，不存在 C 扩展冲突问题。

### Q4. 这个项目的架构为什么这么设计？有没有考虑过其他方案？

**回答策略**：展示架构决策能力。说明你对比了哪些方案、为什么选了当前方案、拒绝了其他方案的具体理由。

**参考回答**：Agent 编排框架方面，我对比了 LangGraph、CrewAI 和 AutoGen。LangGraph 的控制粒度最高（用有向图 StateGraph 精确控制节点间的边和条件路由），状态持久化内置（checkpointer），LangSmith 集成方便调试，生产就绪程度最高。CrewAI 上手最快（2-8 小时），但控制粒度粗（基于角色定义而非图定义），不适合需要精确控制流程的复杂系统。AutoGen 偏学术研究，生产级特性不够。因为本项目需要 3 条独立的流水线和条件路由（更新流中的 retry），LangGraph 是最合适的选择。

### Q5. 你在项目中遇到了哪些技术选型的权衡（trade-off）？

**回答策略**：展示工程判断力，说明你理解每个选择背后的代价。

**参考回答**：有几个典型的权衡。第一，512 字符的 Chunk 大小——太小会丢失上下文导致答案碎片化，太大会引入噪音降低检索精度，512+64 重叠是在语义完整性和检索精度之间的平衡。第二，Neo4j 子图遍历限制 2 跳——超过 2 跳的关系通常太远，引入噪音而非有效信息，而且 Cypher 查询的效率随跳数指数下降。第三，MERGE 语义建节点——优点是避免重复节点，缺点是无法区分不同文档中同名但含义不同的实体（比如两个叫"张三"的人），这是一个精度和去重的 trade-off。

## 五、简历避坑指南

**DO：**

1. **量化结果**："检索 F1 提升 22%"比"检索准确率显著提升"有力 10 倍。面试官能从数字判断你的工作强度和技术含量。
2. **使用动作动词**："设计"、"实现"、"优化"、"解决"——每种动词代表不同的能力维度。设计=架构能力，实现=编码能力，优化=性能思维，解决=攻克难题。
3. **突出技术深度**：不只说"用了 LangGraph"，要说"用 LangGraph StateGraph 实现有向图编排，定义了 3 条独立流水线，更新流实现了条件路由的自动重试"。深度体现在你对工具的掌握程度。
4. **体现架构能力**：说明你的设计决策和理由——"为什么用 MERGE 而不是 CREATE"、"为什么 Chunk 选 512 而不是 256"。这表明你不是在搬砖，而是在做工程判断。
5. **与岗位匹配**：投 AI 岗突出 RAG/Agent/知识图谱/Embedding，投后端岗突出 FastAPI/异步/系统设计/服务编排，投全栈岗加上前端设计。

**DON'T：**

1. **不要堆砌技术词**：写自己真正理解并能展开讲的技术。简历上写"精通 Kafka"但被问到 partition 策略就卡住，比不写更糟糕。
2. **不要夸大数据**：面试官会追问。"准确率 99%"——"你的测试集是什么？"——答不上来就崩了。宁可写"F1 约 85-90%"也不要写一个无法解释的精确数字。
3. **不要写团队成果**：用"我"而不是"我们"。面试官想知道你个人做了什么。
4. **不要写太多细节**：简历是概述，细节留给面试。4-6 个要点够了。
5. **不要把不相关的技术都写上**：选岗位对应的技术栈，AI 岗位不需要强调你会 Docker Compose。
