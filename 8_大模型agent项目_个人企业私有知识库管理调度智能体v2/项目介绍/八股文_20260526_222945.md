## 1. RAG vs 微调

**RAG (Retrieval-Augmented Generation)**：推理时从外部知识库检索相关文档，拼入 Prompt 作为上下文，LLM 基于检索内容生成答案。模型权重不变。

**微调 (Fine-tuning)**：用标注好的领域数据继续训练模型，改变模型权重，让模型学习特定的格式、语气或领域推理方式。

**两者对比**：RAG 适合知识频繁更新、需要引用来源、多知识域共存的场景。微调适合需要模型学习特定格式、语气或推理方式的场景。两者不互斥——可以先微调模型让它学会领域风格，再配合 RAG 获取最新知识。

**本项目**：纯 RAG 方案，没有微调。因为企业文档频繁更新，RAG 的"改知识库即改答案"特性更适合。

```python
# RAG 的核心模式
contexts = await vector_store.search(query, top_k=5)  # 检索
prompt = f"基于以下上下文回答：\n{contexts}\n\n问题：{query}"  # 拼接
answer = await llm.ainvoke(prompt)  # 生成
```

**面试追问**：「什么情况下应该同时用 RAG 和微调？」——当你既需要模型理解特定领域的格式习惯（微调），又需要获取最新的事实信息（RAG）时。

## 2. Agent vs Chain

**Chain**：预定义的固定流程，先 A 后 B 再 C，无论中间结果如何都走固定路径。

**Agent**：具有自主决策能力的 AI 程序，由 LLM 核心、规划能力、记忆、工具使用四个组件构成。根据中间结果动态决定下一步做什么。

**Chain vs Agent 对比**：

| 维度 | Chain | Agent |
|------|-------|-------|
| 执行流程 | 固定、预定义 | 动态、LLM决定 |
| 灵活性 | 低 | 高 |
| 可调试性 | 高 | 低 |
| 适用场景 | 确定性任务 | 需要推理的任务 |
| 类比 | 流水线工人 | 项目经理 |

**本项目**：LangGraph 的 StateGraph 处于两者之间——节点执行顺序是预定义的（有向图），但节点内部 LLM 可以动态决策（如 QAAgent 内部根据意图选择检索策略）。

**面试追问**：「LangGraph 算是 Agent 框架还是 Chain 框架？」——它是有向图的有状态编排框架，可以同时构建 Agent 和 Chain。它的核心价值在于显式控制流和状态持久化。

## 3. 多 Agent 编排模式

**主从模式 (Boss-Worker)**：中央编排者分配任务给 Worker Agent，Worker 汇报结果，编排者汇总决策。

**流水线模式 (Pipeline)**：上游 Agent 的输出是下游 Agent 的输入，串联执行。

**讨论模式 (Joint Discussion)**：多 Agent 平等对话协作，各自提出观点后综合决策。

| 模式 | 特点 | 优点 | 缺点 |
|------|------|------|------|
| Boss-Worker | 中央协调 | 职责清晰 | 编排者瓶颈 |
| Pipeline | 线性传递 | 简单直观 | 不适合并行 |
| Joint Discussion | 平等协作 | 多视角 | 易死循环 |

**本项目**：入库流是纯 Pipeline（parse→extract→store），问答流是单节点（Agent 内部并行检索），更新流是 Pipeline + 条件路由（process→done/retry）。整体最接近 Boss-Worker 模式——LangGraph 编排器充当 Boss。

**面试追问**：「为什么不直接用 CrewAI 这种纯 Boss-Worker 框架？」——LangGraph 的显式有向图比 CrewAI 的角色定义更精准，更适合需要精确控制流程的企业级场景。

## 4. 向量检索的原理与局限

**原理**：将文本通过 Embedding 模型转为高维向量（本项目 384 维），语义相似的文本在向量空间中距离近。检索时用 query 的向量和库中所有向量计算余弦相似度（`cosine_similarity`），返回 top-K。

**ANN (近似最近邻)**：暴力计算所有向量的相似度太慢。ChromaDB 使用 HNSW 算法（分层可导航小世界图），牺牲微小精度换取 10-100 倍的速度提升。

**局限**：
- 只能捕获语义相似性，无法理解结构化关系
- 对精确匹配不敏感（"订单号 #2025-001"和"#2025-001"向量可能不相似）
- 多跳推理无能为力（"张三的老板的公司"需要 3 跳）

**本项目**：用知识图谱补充向量检索的局限——向量覆盖语义，图谱覆盖结构。

**面试追问**：「ANN 会损失多少精度？」——HNSW 的召回率通常在 95-99% 之间（取决于参数配置），损失的 1-5% 可以通过 Rerank 阶段弥补。

## 5. GraphRAG 核心知识

**纯向量检索的局限**：只能捕获语义相似性，无法理解实体间的结构化关系。对多跳推理无能为力。

**GraphRAG 的优势**：保留实体间的显式关系，支持多跳推理（A→B→C 的推理链），提供推理路径使答案可解释性更强。

**本项目的 GraphRAG 实现**（`services/graph_rag.py`）：6 步管线——向量检索→实体链接→子图遍历→路径推理→社区摘要→交叉重排序。权重设计为向量×1.0、子图×1.15、路径×1.25、社区×1.1。

```python
# GraphRAG 交叉重排序的核心逻辑
weight_map = {"vector": 1.0, "subgraph": 1.15, "path": 1.25, "community": 1.1}
for ctx in contexts:
    ctx.score *= weight_map.get(ctx.source_type, 1.0)
# 去重后按加权分降序排序
unique.sort(key=lambda c: c.score, reverse=True)
```

**面试追问**：「GraphRAG 和 Graph+Vector RAG 有什么区别？」——GraphRAG 特指微软提出的方法，包含社区摘要（Community Summary）；Graph+Vector RAG 是泛指融合图谱和向量的检索方法。本项目更接近后者，但也实现了社区摘要。

## 6. CDC 增量更新

**CDC (Change Data Capture)** 的核心思想：不是"定期全量扫描"，而是"监听变更事件并增量处理"。

```
传统方案: 每小时全量扫描 → 全量重解析 → 全量重入库 → ~30分钟
CDC方案:  实时监听变更 → 只处理变化的文件 → 增量更新 → ~30秒
```

**两种 CDC 模式**：
- 文件系统 CDC：Watchdog 监听文件创建/修改/删除事件（基于 OS 的文件系统事件）
- 数据库 CDC：消费 Kafka 中的 Debezium 事件（监听 MySQL binlog / PostgreSQL WAL）

**本项目**：双模式都实现了。Watchdog 模式在 `KnowledgeUpdateAgent.start_watching()`，Kafka 模式在 `start_kafka_consumer()`。差量计算在 `CDCProcessor.compute_diff()`，>30% 变化率判定为大改。

```python
# 差量计算：区分大改和小改
@staticmethod
def compute_diff(before: str, after: str) -> dict:
    change_ratio = len(added | removed) / max(len(before_lines) + len(after_lines), 1)
    return {"change_ratio": round(change_ratio, 4), "is_major_change": change_ratio > 0.3}
```

**面试追问**：「30% 的阈值是怎么定的？」——经验值。低于 30% 通常是局部修改，重做局部即可；超过 30% 说明文档结构有重大变化，全量重建更可靠。具体阈值需要根据实际业务的数据变更模式调整。

## 7. Embedding 模型选型

| 模型 | 维度 | 中文效果 | 成本 |
|------|------|---------|------|
| text-embedding-3-small | 1536 | 中 | API 付费 |
| text-embedding-3-large | 3072 | 高 | API 付费 |
| bge-large-zh | 1024 | 最佳 | 开源免费 |
| text2vec-base-chinese | 384 | 好 | 开源免费 |

**本项目**：用 `text2vec-base-chinese`（384 维，本地 CPU），选它的原因是中文效果好、轻量、零成本。384 维对中小规模文档完全够用，且推理速度快。

**面试追问**：「为什么不直接用 OpenAI Embeddings？」——成本考虑。每次检索都要调用 Embedding API，1000 个文档 × 45 chunks × 1536 维 = 大量 API 调用。本地模型一次性加载后零边际成本。但生产环境如果已有 OpenAI 额度，用 text-embedding-3-small 也是合理选择。

## 8. Chunking 策略

**Chunk 大小的权衡**：

| Chunk Size | 优点 | 缺点 |
|------------|------|------|
| 过小 (128) | 检索精准 | 丢失上下文，答案碎片化 |
| 过大 (2048) | 保留上下文 | 检索噪音大，无关内容多 |
| 推荐 (512) | 平衡 | - |

**本项目**：512 字符 + 64 字符重叠。512 能覆盖大多数段落，64 重叠保证 Chunk 边界处的关键信息（如"他负责微信产品线"的"他"指向上一个 Chunk 最后提到的"张三"）不被截断。

```python
CHUNK_SIZE = 512
CHUNK_OVERLAP = 64
while start < len(text):
    end = start + CHUNK_SIZE
    content = text[start:end]
    # ...
    start = end - CHUNK_OVERLAP  # 重叠滑动
```

**面试追问**：「什么时候用语义分块而不是固定长度分块？」——当文档有明确的章节/段落结构时，语义分块更好（按标题、段落边界切分）。固定长度分块的优点是简单、可预测。本项目用固定长度是因为文档类型多样，语义边界难以统一。

## 9. Neo4j 图数据库基础

**图数据库 vs 关系数据库**：图数据库以节点和边为基本单位，天然适合表达实体间的关系网络。关系数据库用 JOIN 实现多跳查询，3 跳以上的 JOIN 性能急剧下降。图数据库用指针跳转实现关系遍历，跳数增加对性能影响小。

**Cypher 查询语言**：Neo4j 的声明式图查询语言。`MATCH (a)-[r]->(b)` 匹配有向关系，`[*1..N]` 表示 N 跳可变长度路径，`shortestPath()` 计算最短路径。

**本项目的 Neo4j 使用**：每个 Entity 是一个节点（Label: `Entity`，属性: `name/type/description/version/source/timestamp`），每个 Relation 是一条有向边（关系类型动态映射为 WORKS_AT 等大写下划线格式）。3 个索引加速 name/type/source 查询。

```cypher
-- 多跳子图遍历
MATCH path = (start:Entity {name: $name})-[*1..2]-(neighbor)
RETURN start.name, [r IN relationships(path) | type(r)], neighbor.name
LIMIT 50
```

**面试追问**：「Neo4j 和 PostgreSQL 的递归 CTE 有什么区别？」——对于 2-3 跳的关系遍历，两者性能差异不大。但 5 跳以上的图遍历，Neo4j 的指针跳转比 SQL 递归 CTE 的临时表 JOIN 快 10-100 倍。关键差距在于数据模型的天然适配。

## 10. Docker Compose 多服务编排

**多服务编排的核心问题**：服务启动顺序（Neo4j 必须先于 API），健康检查（确认 Neo4j 真正 ready），网络隔离（容器间通过服务名通信）。

**本项目的 docker-compose.yml**：5 个服务——Neo4j（`ports: 7474/7687`）、ChromaDB（`port: 8000`）、Zookeeper + Kafka（`port: 9092`）、API（`port: 8080`，`depends_on: [neo4j, chromadb, kafka]`）。3 个命名卷持久化数据。健康检查监控 Neo4j 状态。

```yaml
neo4j:
  image: neo4j:5.28-community
  healthcheck:
    test: ["CMD", "neo4j", "status"]
    interval: 10s
    timeout: 5s
    retries: 5
```

**面试追问**：「depends_on 能保证 Neo4j 完全启动后再启动 API 吗？」——不能。`depends_on` 只保证容器启动顺序，不保证服务 ready。所以 Neo4j 加健康检查，API 启动后如果 Neo4j 未 ready 会 try-except 容错。

## 11. FastAPI 异步编程

**ASGI vs WSGI**：WSGI 是同步模型（一个请求一个线程），ASGI 是异步模型（一个事件循环处理多个请求）。FastAPI 基于 ASGI，原生支持 `async/await`。

**本项目中的异步使用**：所有 API 端点用 `async def`，LLM 调用用 `await self.llm.ainvoke()`，Neo4j 用 `AsyncGraphDatabase`，LangGraph 用 `ainvoke()`。向量检索和图谱检索在 QAAgent 中虽未显式 `asyncio.gather` 但天然支持并行。

**面试追问**：「是不是所有函数都应该用 async？」——不是。只有 I/O 密集型操作（网络请求、数据库查询、文件读写）适合 async。CPU 密集型操作（Embedding 推理、PDF 解析）用 async 反而阻塞事件循环。所以 Embedding 模型放在子进程而非直接在 async 函数中调用。

## 12. Prompt Engineering 实践

**系统提示词 (System Prompt)**：定义角色和能力边界。本项目的每个 Agent 有独立的 System Prompt，如知识抽取的 `EXTRACTION_SYSTEM_PROMPT` 约 300 字，严格按照 JSON Schema 定义输出格式。

**少样本 (Few-shot) 提示**：在 Prompt 中提供示例。本项目主要使用 Zero-shot（不给示例，靠清晰的 JSON Schema 指令约束输出），因为格式定义清楚时 Zero-shot 已经足够。

**输出约束**：所有 LLM 输出要求返回纯 JSON（"只返回 JSON，不要包含其他文字"），并做容错解析（去掉 ``` 包裹 → `json.loads()` → JSONDecodeError 兜底返回空结构）。

```python
# JSON 输出容错
cleaned = resp.content.strip()
if cleaned.startswith("```"):
    cleaned = cleaned.split("\n", 1)[1].rsplit("```", 1)[0]
data = json.loads(cleaned)
```

**面试追问**：「为什么不用 LangChain 的 Output Parser？」——容错性。LangChain 的 OutputParser 对格式偏差容忍度低，LLM 偶尔多一个逗号或少一个引号就会解析失败。自己写容错逻辑可以处理更多边界情况。

## 13. 子进程隔离与进程间通信

**为什么需要子进程隔离**：PyTorch 和 ChromaDB 的底层 C 扩展在 Python asyncio 事件循环中不是线程安全的。在主进程中直接调用会导致 segfault——整个进程直接崩溃，无法 try-except 捕获。

**本项目的方案**：将 sentence-transformers 加载到独立子进程，主进程通过 `multiprocessing.Queue` 通信。子进程跑 `_worker_process` 函数，加载模型后进入消息循环等待 encode 请求。主进程的 `EmbeddingClient` 是单例代理。

```python
# 子进程消息循环
while True:
    msg_id, texts = request_queue.get(timeout=1)
    vectors = model.encode(texts).tolist()
    response_queue.put((msg_id, vectors))
```

**面试追问**：「为什么不用 gRPC 或 HTTP 做进程间通信？」——multiprocessing.Queue 是本地进程间最快的通信方式（共享内存），gRPC/HTTP 有序列化和网络开销。子进程是本地的，不需要网络通信的开销。

## 14. 容错与降级策略

**分层容错**：Agent 层的 try-except 保证局部失败不影响全局；服务层的静默跳过保证一个数据库不可用时其他数据库正常工作；编排层的条件路由实现失败自动重试。

**降级策略**：PDF 解析失败→LLM 视觉兜底；向量库不可用→跳过向量存储，只写图谱；Embedding 子进程崩溃→`DISABLE_LOCAL_EMBEDDINGS=1` 开关切换到 API Embedding。

**死信队列**：增量更新中 `should_continue` 条件判断失败任务 → 进入 retry 节点重试 → 重试失败后仍然返回结果（标记 success=False），不阻塞。

**面试追问**：「为什么不引入熔断器？」——当前的错误率不高，try-except + 静默降级已足够。如果生产环境中错误率上升（如外部 API 频繁超时），应该引入熔断器（如 `tenacity` 或 `circuitbreaker`）防止级联故障。

## 15. 数据库索引设计

**Neo4j 索引**：3 个索引——`Entity.name`（查找特定实体）、`Entity.type`（按类型过滤）、`Entity.source`（按来源删除，支持增量更新）。索引类型是 Neo4j 的 B-tree 索引，支持等值查询和前缀搜索。

**索引选择逻辑**：name 和 type 是最高频的查询条件，source 是增量更新时的必需过滤条件。没有对 description 建索引——它是模糊查询（CONTAINS），B-tree 索引对模糊查询无效，需要全文索引。

```cypher
CREATE INDEX IF NOT EXISTS FOR (n:Entity) ON (n.name)
CREATE INDEX IF NOT EXISTS FOR (n:Entity) ON (n.type)
CREATE INDEX IF NOT EXISTS FOR (n:Entity) ON (n.source)
```

**面试追问**：「为什么不对关系类型建索引？」——Neo4j 的关系类型本身就是一等公民（存储级别），查询 `()-[r:WORKS_AT]->()` 直接走关系类型过滤，不需要额外索引。

## 16. 多模态 RAG

**挑战**：PDF 中的图片、表格、流程图无法被传统的文本 Embedding 检索覆盖。

**"万物转文本"策略**：所有模态先转为文本表示，再做统一 Embedding。图片→OCR 文字 + LLM 视觉描述，表格→"列名:值"文本表示，PDF→PyPDF2 文字提取 + LLM 视觉兜底。

**检索时的模态加权**：`services/multimodal.py:38-44`，不同模态的来源文档在检索时施加不同权重——text×1.0（最可靠）、pdf×0.95（可能有解析噪音）、table×0.9（结构化转文本可能丢失关系）、image×0.85（最不确定，LLM 对图片的理解可能有偏差）。

```python
MODALITY_WEIGHTS = {
    "text": 1.0, "markdown": 1.0,
    "pdf": 0.95, "table": 0.9, "image": 0.85,
}
```

**面试追问**：「为什么不直接对图片做 Embedding（如 CLIP）？」——CLIP 对自然图片效果好，对文档中的图表/截图效果有限。"万物转文本"策略更通用——转为文本后所有下游处理都统一，不需要为每种模态维护单独的检索系统。
