## 概述

文档入库流程解决的核心问题：**如何将一份任意格式的文档，自动转化为可供AI检索和推理的结构化知识**。

这是一个"非结构化→半结构化→结构化"的三阶段转换过程：

```
PDF/图片/表格/文本 → [解析] → 文本块序列 → [抽取] → 实体+关系 → [存储] → 向量库 + 知识图谱
```

与其他系统不同的是，本项目的入库流程不走"全部存入向量库就完事"的捷径——它同时维护向量索引和知识图谱两套互补的知识表示，为后续的GraphRAG混合检索奠定基础。

**流程对比**：

| 维度 | 传统RAG入库 | 本项目入库 |
|------|-----------|-----------|
| 文档格式 | 通常只支持纯文本 | 自动适配PDF/图片/表格/文本/Markdown |
| 知识表示 | 只有向量（语义相似度） | 向量 + 知识图谱（语义+结构） |
| 分块策略 | 固定长度或固定句子 | 512字符 + 64重叠滑动窗口 |
| 更新方式 | 全量删除重建 | 增量更新（按doc_id精确删除替换） |
| 失败处理 | 通常整体失败 | 逐环节容错，部分成功仍返回结果 |

## 读取/加载机制

### 文档类型识别

项目采用**扩展名映射表**自动路由解析器，无需人工指定文档类型：

```python
SUPPORTED_EXTENSIONS = {
    ".pdf": DocType.PDF, ".png": DocType.IMAGE,
    ".jpg": DocType.IMAGE, ".csv": DocType.TABLE,
    ".xlsx": DocType.TABLE, ".txt": DocType.TEXT,
    ".md": DocType.MARKDOWN, ...
}
```

识别失败时兜底为`DocType.UNKNOWN`，走纯文本解析路径。

### 解析器选择策略

不同于常见方案中所有格式用同一个大而全的解析器处理，本项目为每种格式写了独立的解析函数，优点有两个：
1. 每个解析器可以针对性优化（比如PDF专门做了PyPDF2→LLM视觉的降级链）
2. 新增格式只需加一个`_parse_xxx()`方法和一行注册，不影响现有逻辑

### 缓存策略

项目当前没有语义缓存（如相同文件跳过重复解析）。但因为SHA256哈希机制已就位（`_make_doc_id`），很容易在此基础上加一层检查：入库前先查`doc_id`是否已存在，存在则跳过。

### 配置来源

所有路径、超时、模型选择通过`config/settings.py`的`BaseSettings`从`.env`加载，19个配置项全部有默认值。关键配置：
- `CHUNK_SIZE=512, CHUNK_OVERLAP=64`（硬编码在Agent类属性中）
- `BATCH_SIZE=5`（知识抽取每次LLM调用最多处理5个chunk）
- `UPLOAD_DIR="./uploads"`（文件上传存储目录）

## 阶段1：文档解析 — 多模态统一入口

**为什么不用单一的Unstructured库一把梭？**

Unstructured确实支持多格式，但有两个问题：
1. PDF中的嵌入图片/表格需要额外处理，Unstructured对这些元素的理解能力有限
2. 中文文档的扫描件OCR是刚需，Unstructured对中文OCR支持不稳定

所以项目采取了"专用解析器+LLM视觉兜底"的双层策略。

### 决策点：PDF解析的分层降级

```
尝试PyPDF2提取文字
  ├─ 成功且有内容 → 直接使用（最快，零LLM成本）
  └─ 失败或为空 → 降级到LLM视觉理解
                    ├─ pdf2image渲染为图片(dpi=150)
                    ├─ base64编码发送给DeepSeek多模态API
                    └─ LLM描述页面所有内容 → 转为文本
```

这里有一个容易被忽略的细节：`_pdf_vision_fallback`只处理前5页（`last_page=5`）。这是有意为之——如果一份PDF前5页全部提取失败，通常意味着整份文档都是扫描件，5页采样已经足够LLM理解文档性质；如果前5页中大部分有文字，不会触发降级。

### 决策点：图片的双通道理解

图片不走"OCR或LLM视觉二选一"，而是两者都做，结果叠加：
- OCR擅长提取精确文字（但可能漏掉上下文）
- LLM视觉擅长描述图表、结构、整体含义（但文字可能不准）
- 两者叠加 → 同一个图片产生两段互补的文本chunk

### 决策点：表格的结构保持

CSV和Excel解析后并没有直接存原始行列数据，而是转为"列名: 值"的自然语言表示：
```
表头: 姓名 | 职位 | 部门
姓名: 张三 | 职位: CEO | 部门: 技术部
姓名: 李四 | 职位: CTO | 部门: 技术部
```
这样做的原因是后续的Embedding模型和LLM都是为自然语言优化的，结构化数值直接嵌入效果很差。"列名:值"的表示保留了语义。

## 阶段2：知识抽取 — 非结构化到结构化

**为什么用LLM做抽取而不是训练专用NER模型？**

三个原因：
1. 企业内部文档的实体类型是开放的——今天是"张三(研发工程师)"，明天是"Q3营收(财务指标)"，专用NER的标签集很难覆盖
2. 关系抽取需要理解上下文，这是LLM的强项
3. 放弃了规则/模型组合的维护成本——修改Prompt比重新训练模型快得多

代价是每chunk一次LLM调用，处理1000个chunk需要1000次API请求。BATCH_SIZE=5的设计是为了减少LLM调用的总次数（每次调用处理5个chunk），但单次调用的输入更长，需要在Token消耗和请求次数之间取得平衡。

### 决策点：去重的粒度

实体去重只用`(name, type)`：
- "张三"在chunk_1被识别为Person
- "张三"在chunk_5也被识别为Person
- → 只保留第一个，后续丢弃

这意味着如果同一份文档中出现了两个"张三"但类型不同（比如一个Person一个Concept），都会被保留。这是合理的选择——同名异义的情况在企业文档中确实存在。

关系去重用完整的`(head, relation, tail)`三元组去重，保证图谱中不出现完全相同的边。

### 决策点：confidence的使用

relation的confidence由LLM评估并返回（0-1之间的浮点数），但在当前的Neo4j存储中，confidence只是作为关系属性存储，没有在检索阶段作为权重因子。这是一个可以改进的点——在GraphRAG的cross_rerank中把relation confidence纳入评分，让LLM标注的高置信度关系获得更高的检索权重。

## 阶段3：双写存储 — 向量和图谱并行

**为什么同时维护向量库和知识图谱？**

纯向量的局限：向量检索基于语义相似度，它理解"苹果"和"水果"相似，但不理解"苹果公司的CEO是谁"——因为"CEO"和"苹果公司"之间的关系不是语义近邻，而是结构化关系。

纯图谱的局限：图谱擅长关系推理，但对长文本的语义理解弱——"这个产品的核心竞争力是什么"这种问题查Cypher很难回答，因为"核心竞争力"是语义概念，不会作为实体存在。

两者互补：向量找到语义相关的内容，图谱提供实体间的结构化关系，融合后既不失语义广度，又有关系深度。

### 决策点：为什么用MERGE而不是CREATE

```cypher
MERGE (e:Entity {name: $name})  -- 以name为键，存在则更新，不存在则创建
ON CREATE SET ...
ON MATCH SET ...
```

选用MERGE是为了支持增量更新。CREATE会重复建节点，导致图谱中同一个"张三"有多个节点，检索时关系断裂。MERGE保证同名实体只有一个节点，多次入库只会更新属性。

### 决策点：ChromaDB的async安全措施

这是项目中最"工程化"的设计决策。`vector_store.py`中随处可见对ChromaDB C扩展的防御性处理：
- 所有ChromaDB操作通过线程池执行（`run_in_executor`）
- 向量计数手动维护而非调用ChromaDB API
- `embeddings_available`属性做防御性检查
- Embedding模型运行在独立子进程（`embedding_worker.py`）

根本原因是ChromaDB的底层C库（基于hnswlib）在Python asyncio事件循环中存在稳定性问题。在开发和演示环境中这些措施保证了系统不会因为一次向量操作就整个进程crash。在生产环境建议切换到PGVector或Milvus。

## 关键数据结构

```python
# ── 文档层 ──
@dataclass
class DocumentChunk:
    content: str                        # 文本内容
    doc_id: str                         # SHA256前16位 "a3f8c2d1e4b5..."
    chunk_index: int                    # 0, 1, 2, ...
    doc_type: DocType                   # PDF / IMAGE / TABLE / TEXT / MARKDOWN
    metadata: dict[str, Any]            # {"source": "/path/to/file.pdf", "char_start": 0, ...}
    embedding: list[float] | None       # 384维向量，延迟计算

# ── 知识层 ──
@dataclass
class Entity:
    name: str                           # "张三"
    type: str                           # "Person"
    description: str                    # "腾讯公司CEO，负责微信产品线"
    properties: dict[str, Any]          # {"年龄": 45, "入职时间": "2015"}

@dataclass
class Relation:
    head: str                           # "张三"
    relation: str                       # "works_at"
    tail: str                           # "腾讯"
    confidence: float                   # 0.95

@dataclass
class ExtractionResult:
    entities: list[Entity]              # 本chunk提取的实体
    relations: list[Relation]           # 本chunk提取的关系
    events: list[KnowledgeEvent]        # 本chunk提取的事件
    source_chunk_id: str                # "a3f8c2d1#chunk-3"

# ── CDC层 ──
@dataclass
class CDCEvent:
    event_id: str                       # 事件唯一ID
    source_type: str                    # "filesystem" | "database"
    operation: str                      # "INSERT" | "UPDATE" | "DELETE"
    resource_path: str                  # 受影响的文件/表
    diff: dict[str, Any] | None         # {"change_ratio": 0.45, "is_major_change": True}
```

### 关键决策点复盘

1. **第2页表格丢失**：PyPDF2无法提取表格文字，且因为第1页有内容所以没触发视觉兜底。这意味着当前实现下，嵌入PDF中的表格可能被漏掉。改进方向：检查每页提取的文字，对空白页单独触发视觉兜底。

2. **关系类型"负责"**：LLM产出的关系类型不在预定义的8种列表中（belongs_to, works_at, ...），而是自由生成了"负责"。这实际上是个特性而非bug——预定义列表只是指导，LLM可以根据文本灵活生成关系类型。

3. **只有一个chunk**：50字符的文本远小于CHUNK_SIZE=512，所以只切出一个chunk。对于2页的PDF产出45+chunks是更典型的场景。

## 关键设计模式

| 设计模式 | 在项目中的应用 | 为什么这样用 |
|---------|---------------|------------|
| **策略模式** | `DocParserAgent`根据文件类型路由到不同的`_parse_xxx()`方法 | 每种格式的解析逻辑差异很大，策略模式避免在parse()中出现巨大的if-else |
| **管道模式 (Pipeline)** | LangGraph的parse→extract→store_vectors→store_graph串行节点 | 上下游数据依赖明确，线性的管道最直观 |
| **模板方法** | `_extract_from_chunk()`定义抽取骨架，`_parse_response()`处理JSON解析细节 | 抽取流程固定（LLM调用→JSON解析→构造对象），解析逻辑可变 |
| **单例模式** | `config/settings.py`全局`settings`实例；`get_embedding_client()`的`_embedding_client`单例 | 配置和Embedding模型全局只需一份，避免重复加载 |
| **代理模式** | `services/embedding_worker.py`的`EmbeddingClient`作为子进程Embedding的代理 | 隔离PyTorch C扩展，Main进程不直接持有模型引用 |
| **观察者模式** | `KnowledgeUpdateAgent.start_watching()`使用Watchdog的`FileSystemEventHandler` | 文件系统变更事件的异步通知 |

## 常见问题排查

### Q1: 入库后问答找不到信息，但文档明明有

**检查清单**：
1. 入库响应的`chunks_count`是否为0？→ 检查文件格式是否在`SUPPORTED_EXTENSIONS`中
2. 入库响应的`entities_count`是否为0？→ 检查LLM API是否正常（看日志中LLM调用是否成功）
3. PDF是否主要是扫描件？→ 检查是否触发了`_pdf_vision_fallback`，未触发说明text提取非空但内容质量差
4. 文档是否包含大量图片/表格？→ 图片的OCR和LLM描述可能不完整

### Q2: 入库后Neo4j中看不到节点

**检查清单**：
1. Neo4j容器是否运行？→ `docker-compose ps neo4j`
2. 密码是否正确？→ 检查`.env`中`NEO4J_PASSWORD`
3. API日志是否有Neo4j连接错误？→ 入库流程在`store_graph`失败时会静默吞下异常
4. 尝试直接访问Neo4j Browser → `http://localhost:7474` 查看数据

### Q3: ChromaDB segfault

**检查清单**：
1. 是否在async环境下调用？→ 项目已做线程池隔离，但仍然建议设置`DISABLE_LOCAL_EMBEDDINGS=1`跳过本地Embedding
2. PyTorch版本是否和系统兼容？→ 尝试重新安装`pip install --force-reinstall sentence-transformers`

### Q4: 知识抽取结果质量差（实体不准确、关系混乱）

**检查清单**：
1. LLM模型能力是否足够？→ DeepSeek-Chat的JSON格式遵循度略弱于GPT-4，可尝试切换到更强模型
2. Chunk是否太小或太大？→ 当前512字符，如果文档信息密度高可以适当增大
3. JSON解析是否频繁失败？→ 查看日志中`JSONDecodeError`的频率，高则说明LLM输出了非标准JSON

## 性能优化建议

| 方向 | 当前状态 | 优化方案 |
|------|---------|---------|
| 文档解析 | 单文件串行 | 多文件并行解析（asyncio.gather），10个文件同时parse_batch |
| 知识抽取 | 每chunk一次LLM调用 | 批量Prompt：一次LLM调用处理5-10个chunk，减少API往返 |
| 向量存储 | 手动计数，未实际embed+store | 恢复实际Embedding调用，或用API Embedding服务替代本地模型 |
| 图谱存储 | 逐个实体/关系串行写入 | 批量Cypher：UNWIND一次性写入多个实体/关系 |
| 大PDF处理 | 全量读取 | 流式解析，边解析边抽取，不等待全部chunk完成 |
| 增量更新 | 全文档重新解析 | 已实现差量分析(>30%大改)，但实际处理仍是全量重做 |
